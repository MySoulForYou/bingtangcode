# flow-guide.md · bingtangCode 记忆系统

面向小白。读完你能理解：bingtangCode 如何通过双层记忆架构（工作记忆与长期记忆）让 Agent 拥有记忆、多层级项目指令（`BINGTANGCODE.md`）的优先级合并与安全沙箱机制、基于 JSONL 的追加写会话持久化与免元数据属性动态计算原理、崩溃恢复时的消息链校验截断与 6 小时暂停提醒插入机制、以及后台异步自动笔记提炼与 `MEMORY.md` 索引精简更新原理。

---

## 一、记忆系统端到端流程：从会话启动到记忆沉淀

为了帮助你建立全局观，以下是 `bingtangCode` 从会话启动载入、正常交互落盘、启动崩溃恢复校验，直至会话结束后触发异步提炼更新索引的完整生命周期流程：

```mermaid
graph TD
    A[MewCode 启动] --> B[1. 一致性加载静态指令与记忆索引]
    B --> B1["BINGTANGCODE.md 三层合并 (优先级排序)"]
    B --> B2["MEMORY.md 用户与项目级索引提取"]
    B1 & B2 --> B3[拼装并一次性注入 System Prompt]
    
    A --> C[2. 扫描与恢复会话历史]
    C --> C1[展示历史会话列表或新建会话]
    C1 --> C2[解析 JSONL 并运行 ValidateMessageChain 校验]
    C2 --> C3[截断不完整 tool_calls 并检测 6 小时时间差注入提醒]
    
    B3 & C3 --> D[3. 初始化并开启会话循环]
    D --> E[4. 交互中分阶段追加落盘 (WAL)]
    E --> F[5. 对话自然结束 / 空闲]
    F --> G[6. 异步发起 LLM 提炼四类笔记]
    G --> H[7. 模型自主合并/淘汰并更新 MEMORY.md 索引]
```

## 二、记忆系统解决什么问题

### 之前的问题
- **每次失忆**：每次新开终端或切换会话，Agent 对当前项目的技术栈、开发规范（如“使用 any 代替空接口”）和用户的个人偏好一无所知，需要用户在 Prompt 中反复重复说明。
- **崩溃导致协议报错**：由于网络或终端意外中断，会话文件被不完整截断，存在未结对的 `tool_use`，再次加载发送给 API 时，直接触发大模型 API 校验 400 报错，会话无法继续。
- **上下文缓存失效（Prompt Cache Miss）**：如果记忆和指令随对话轮次高频更新或动态注入到历史中，会导致模型头部 Prompt 发生变化，破坏提示词缓存命中率，带来高昂的 Token 费用和高昂的响应延迟。

### 现在的能力

| 能做 | 不能做 |
|------|--------|
| 项目级与用户级三层指令优先级加载合并 | 本地向量检索（RAG）与语义库相似度计算 |
| `@include` 深度限制（5 层）与循环引用环路检测 | 团队/多端记忆网络云同步 |
| 安全沙箱隔离（超出项目/配置根边界自动静默跳过并替换为警告注释） | 自动在会话运行中途实时修改并热加载指令（为最大化缓存命中率，运行中保持静态） |
| JSONL 追加写持久化（$O(1)$ 写入，崩溃只丢最后一行） | 自动清理落盘的临时工具结果大文件（冷备文件保留供 ReadFile 追溯） |
| 免元数据扫描（通过扫描 JSONL 动态计算标题、时间与会话数） | 对 `MEMORY.md` 索引文件的物理强制截断 |
| 工具链完整性校验（有呼必有应，截断异常不完整 tool_calls） | |
| 6 小时长暂停时间跨度 user 消息自动注入 | |
| 启动时异步清理 30 天前过期会话 | |
| 异步大模型提炼四类笔记并写入 `<type>_<short_slug>.md` | |
| `MEMORY.md` 大小（200行/25KB）由模型在更新时自动合并与淘汰 | |

## 三、关键设计原理与实现细节

以下是记忆系统各项当前能力的具体实现逻辑与核心原理分析：

### 1. 多层级项目指令合并载入
* **实现类**：`InstructionLoader`
* **实现原理**：
  * 程序在启动时，会依次查找三个特定路径的文件：
    1. 项目根目录下的 `BINGTANGCODE.md`（项目共享级，最高优先级）；
    2. 项目根目录下的 `.bingtangcode/BINGTANGCODE.md`（项目私有级，中等优先级）；
    3. 用户个人主目录下的 `~/.bingtangcode/BINGTANGCODE.md`（用户偏好级，最低优先级）。
  * 每一个存在的文件都会被读入内存，并交由 `@include` 语法解析器进行内部的文档展开。
  * 最终，这三部分解析后的内容，会使用 `---` 作为分隔符进行拼接。高优先级的段落排列在输出文本的最前面，以让 LLM 在首位注意力机制下优先遵循项目规范。

### 2. `@include` 嵌套深度限制与环路检测
* **实现方法**：`InstructionLoader.parseFile(...)`
* **实现原理**：
  * **深度限制**：解析器在递归展开文件时，会在参数中传递一个 `depth` 计数器。当且仅当嵌套层级超过 `MAX_DEPTH = 5` 时，递归强行截止，并在对应引用位置输出警告 `<!-- @include 嵌套层级超出限制，已跳过 -->`。
  * **环路防死锁**：使用 `Set<Path> visited` 记录当前引用路径中所有已访问过文件的规范化绝对路径。在处理下一个 `@include` 时，若发现该路径已存在于 `visited` 集合中（说明存在 A $\rightarrow$ B $\rightarrow$ A 循环引用），则拦截递归，输出 `<!-- @include 检测到循环引用，已跳过: <relative_path> -->`。

### 3. 安全沙箱隔离（路径逃逸拦截）
* **实现方法**：`InstructionLoader.parseFile(...)`
* **实现原理**：
  * **边界划分**：项目级指令的合法根边界是 `<project_root>`，用户级指令的合法根边界是 `~/.bingtangcode/`。
  * **沙箱判定**：当解析器读取到 `@include <path>` 时，会将该相对路径与当前正在解析的文件目录进行拼接，生成规范化的绝对路径 `targetPath.toAbsolutePath().normalize()`。
  * 接着，程序检查该绝对路径是否以合法的根边界路径为前缀（`targetPath.startsWith(boundaryPath)`）。
  * 如果判定为超出边界（即路径逃逸，如试图读取 `../../etc/passwd`），程序不会抛出异常中断整个系统的启动，而是**静默跳过**，并在输出的文档中留下专门的 HTML 警告注释：`<!-- @include 路径超出允许范围，已跳过: <path> -->`。

### 4. JSONL 追加写持久化（WAL 机制）
* **实现类**：`SessionSerializer` 与 `SessionPersister`
* **实现原理**：
  * **格式转换**：`SessionSerializer.serialize` 负责将 Message 对象转换为符合 JSONL 格式的 JSON 字符串。
    * 如果消息是 `SYSTEM`、`USER`（纯文本），直接映射为 `"system"`、`"user"` 角色。
    * 如果消息中含有工具结果（`toolResults`），则会为集合中的每一个结果输出一行角色为 `"tool_result"` 的 JSON。
    * 如果消息是 `ASSISTANT` 且含有工具调用（`toolCalls`），则将其 `content` 和 `tool_use` 合并为 JSON 数组（Content Blocks）存储在 `"content"` 字段中。
  * **追加落盘**：`SessionPersister.appendMessage` 以追加模式 `StandardOpenOption.APPEND` 打开会话对应的 `<sessionId>.jsonl` 文件，实现 $O(1)$ 的磁盘快速写入。
  * **落盘时机**：在 `DialogueManager.doRound` 执行的核心节点（收到用户输入、大模型发出工具请求、工具跑完结果、以及最终生成文本回复）完成后，立即触发单行追加写入。这保证了已发生的状态在程序崩溃时随时可查，符合 Write-Ahead Logging 思想。

### 5. 免元数据扫描计算属性
* **实现方法**：`SessionPersister.listSessions()`
* **实现原理**：
  * **无索引设计**：系统不维护中央索引文件，以防状态漂移。
  * **属性反算**：
    * **创建时间**：直接截取 `.jsonl` 文件名的前 15 位字符（`yyyyMMdd-HHmmss`），使用 `SimpleDateFormat` 进行时间还原。
    * **消息数**：使用 `Files.readAllLines` 读取文件的总行数（由于 JSONL 每一行为一条记录，行数即代表记录数）。
    * **最后活跃时间**：直接定位到文件的最后一行（最后一条 Message 记录），使用 Jackson 解析并提取其中的 `"ts"` 字段作为最后活跃时间戳。
    * **标题计算**：按顺序扫描文件的每一行，找到第一个角色为 `"user"` 且内容为字符串的行，提取其 content。若长度大于 50，则截取前 47 个字符并加上 `...` 组成标题；若小于等于 50，则直接使用。

### 6. 工具链完整性校验（ValidateMessageChain）
* **实现方法**：`SessionRecovery.validateMessageChain(...)`
* **实现原理**：
  * **协议拦截**：当程序启动载入历史时，如果 JSONL 文件的尾部存在未完成的工具调用（即只有大模型发出了 `tool_use`，但还没来得及执行或写入结果系统就崩溃了），直接发送给 API 会因协议不完整而报错 400。
  * **回滚截断**：`validateMessageChain` 从头到尾遍历 JSONL 记录。遇到 `assistant` 的 `tool_use` 时，将工具 ID 加入 `pendingToolUses` 集合；遇到 `tool_result` 时，将对应的工具 ID 从集合中移出。
  * 当且仅当 `pendingToolUses.isEmpty()` 时，证明到目前为止所有的工具请求都得到了完备的回应，记录下当前合法索引 `lastValid = i + 1`。
  * 循环结束后，丢弃掉索引在 `lastValid` 之后的全部消息，使会话历史强制回滚到最后一个工具调用完备的正常状态。

### 7. 6 小时长暂停时间跨度提示注入
* **实现方法**：`SessionRecovery.injectTimeGapReminders(...)`
* **实现原理**：
  * 读档时，程序遍历相邻的两条 JSONL 记录（`prev` 和 `curr`）。
  * 如果它们的时间戳差值 `curr.ts - prev.ts >= 6 小时`（`6 * 60 * 60 * 1000` 毫秒），说明本会话曾被长期闲置。
  * 此时，程序会在它们之间插入一条临时的 `USER` 提醒消息，时间戳置为 `prev.ts + 1`，话术严格按照 F21.5 规范：
    `"[系统提示] 本会话已暂停 <duration>。部分上下文可能已过时，如需最新信息请重新读取相关文件。"`（其中 `<duration>` 换算为 `X 小时 Y 分钟` 格式）。这能让 Agent 在继续对话前，重新对文件时效性保持警惕。

### 8. 启动时异步清理 30 天前过期会话
* **实现方法**：`SessionPersister.startAsyncCleanup()`
* **实现原理**：
  * 在 BingtangCode 启动装配阶段，`Main.java` 会在主线程之外通过 `CompletableFuture.runAsync(...)` 启动一个后台异步清理任务。
  * 该任务扫描会话目录下的 `.jsonl` 文件，提取文件名中的创建时间戳并与当前系统时间戳作差。
  * 如果差值大于 **30 天**（30L * 24 * 60 * 60 * 1000 毫秒），说明会话已过期。系统在后台调用 `Files.deleteIfExists` 将该文件静默删除，并级联递归清理其下对应的工具结果物理子目录，完全不阻塞主进程启动。

### 9. 异步自动笔记提炼与写入
* **实现方法**：`AutoMemoryCollector.triggerAsyncMemoryCollection(...)`
* **实现原理**：
  * **生命周期拦截**：在 `AgentLoop` 中，每当 `result.completed()` 为 `true`（且不包含工具调用）时，说明这一轮 Agent 对话已自然停下。
  * **异步执行**：程序在后台调用 `CompletableFuture.runAsync` 启动提炼任务。
  * **模型提取**：构造包含系统指令的 prompt 喂给 LLM。模型负责从对话中提取“用户偏好”、“纠正反馈”、“项目知识”与“参考资料”四类笔记。
  * **分类写入**：大模型在返回的 JSON 中输出 `slug`（全小写、下划线分隔，如 `terse_replies`）。程序将其拼装为带 Frontmatter 的 Markdown：
    * `user_preference` 和 `correction_feedback` 写入用户级配置目录 `~/.bingtangcode/memory/` 下的 `user_preference_terse_replies.md` 等文件。
    * `project_knowledge` 和 `reference_material` 写入项目级目录 `.bingtangcode/memory/` 下的 `project_knowledge_xxx.md` 等文件。

### 10. `MEMORY.md` 索引大模型合并与淘汰
* **实现类**：`AutoMemoryCollector.java`
* **实现原理**：
  * **无硬编码裁剪**：系统本身不写任何强制物理截断代码（不主动裁剪）。
  * **全量委托**：在调用 LLM 提炼笔记时，程序读取磁盘上最新的项目级和用户级 `MEMORY.md` 内容全量传入 Prompt，并附加了强约束条件：“*行数严格限制在 200 行内，物理体积在 25KB 内；若新记忆写入导致超限，请在大模型端自行合并、合并同类项或淘汰最旧条目*”。
  * 大模型在返回的 JSON 中直接输出更新完成后的 `updatedProjectIndex` 和 `updatedUserIndex` 全量文本。程序收到后，直接覆盖写入对应的 `MEMORY.md`。

## 四、缓存稳定性的设计权衡（Prompt Cache Optimization）

为了确保 BingtangCode 的费用控制和极速响应，记忆系统做出了“静态载入”的重大设计妥协：

* **静态初始化**：
  项目指令（`BINGTANGCODE.md`）和记忆索引（`MEMORY.md`）只在会话启动装配阶段通过 `Main.java` 一次性构建，并写入 `Role.SYSTEM`。
* **为什么中途不更新 System Prompt？**
  如果每一轮对话大模型产生了新笔记后，我们都去修改 System Prompt，会导致大模型的 **Prompt Cache（提示词缓存）在每一轮都失效**。每一次发问都得重新给整篇系统提示词和历史付全款 Token 费，延迟也会暴涨。
* **优雅的机制**：
  新产生的笔记会悄悄在后台写入磁盘。当用户重启 BingtangCode 或开启新会话时，这些新记忆才会被重新加载并作为新的 System Prompt 注入。这非常符合人类在睡眠中巩固记忆的认知规律，也是当前兼顾**“实时性”、“缓存效率”与“费用开销”**的最优解。

## 五、核心类与文件说明

| 类名与路径 | 职责 |
|------|--------|
| **InstructionLoader**<br>`core/InstructionLoader.java` | **指令加载核心**。多层级加载 `BINGTANGCODE.md`，执行 `@include` 深度跟踪（限 5 层），visited 防环路，以及项目/用户目录的安全沙箱边界拦截。 |
| **SessionSerializer**<br>`core/SessionSerializer.java` | **会话序列化中心**。处理 Message 记录到 JSONL 字符串的一对多映射，并负责将 JSONL 记录列表还原合并为 Dialogue 历史。 |
| **SessionPersister**<br>`core/SessionPersister.java` | **会话落盘存储服务**。管理追加落盘写入、扫描会话列表计算动态标题及消息数，并在启动时启动异步后台清理 30 天过期会话。 |
| **SessionRecovery**<br>`core/SessionRecovery.java` | **会话校验与恢复**。执行坏行跳过、无结果工具截断（`ValidateMessageChain`）、检测 6 小时长暂停并注入用户提醒，以及超限自动压缩。 |
| **AutoMemoryCollector**<br>`core/AutoMemoryCollector.java` | **异步记忆收集服务**。监听 Loop 完成事件，开启后台线程让 LLM 提炼四类笔记写入 Markdown，并由 LLM 自行合并/淘汰维护不超过 200 行/25KB 的 `MEMORY.md` 索引。 |
| **DialogueManager**<br>`core/DialogueManager.java` | 在构造时完成指令和记忆的一次性装配，并在 `doRound` 各核心节点立即向 Persister 追加写入 Message 对象以保障 WAL。 |
| **MemorySystemEndToEndTest**<br>`src/test/java/com/bingtangcode/core/MemorySystemEndToEndTest.java` | **端到端集成测试**。完整覆盖安全沙箱路径越权、嵌套检测、追加落盘、动态标题列表计算、有呼无应截断以及 6 小时长暂停提示注入。 |
