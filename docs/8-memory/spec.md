# Specification: BingtangCode 记忆系统

本规范定义了 BingtangCode 终端 AI 编程助手的记忆系统，通过多层级指令加载、会话持久化与自动笔记更新，打破单次会话的“失忆”问题，建立长短期记忆分层管理机制。

## 背景
大语言模型本身没有跨会话的记忆能力，导致 Agent 在每次新启动时都处于“失忆”状态。为了让 AI 助手能够越用越懂用户，同时在网络中断或程序崩溃后能够平滑恢复上下文，需要设计一套轻量、稳定且安全的双层记忆（工作记忆与长期记忆）管理系统。

## 目标用户
需要高频使用终端 AI 编程助手进行复杂、跨会话开发，且对数据安全性与本地部署便利度有较高要求的软件工程师。

## 能力清单

### 1. 多层级项目指令合并载入
- 系统应能从用户个人主目录、本地私有配置目录以及项目根目录三个层级，合并载入多份手写的项目指令说明。
- 指令在载入拼接时应支持优先级规则，高优先级的指令内容必须排列在最前面，以确保在发生规则冲突时优先被模型遵循。

### 2. 模块化指令解析与安全沙箱
- 项目指令文件应支持使用特定标记在文档内引用其他外部 Markdown 文件。
- 引用解析器必须具备防环路检测能力，防止无限循环引用。
- 引用解析器必须限制最大的嵌套引用深度。
- 引用解析器必须实施安全沙箱隔离，拦截任何企图跳出其所属根目录范围的越权路径。若发生越权引用，系统不应中断执行，而是以特殊注释替代该处内容。

### 3. 会话历史追加持久化
- 对话交互记录必须以文本行形式进行追加写入，确保写入操作高效且崩溃安全。
- 系统在重新加载会话时，能够识别并跳过格式损坏的数据行。

### 4. 无元数据状态的管理
- 系统不单独维护会话索引文件，所有会话属性（包括会话 ID、标题、消息数等）必须在扫描目录和解析数据文件时动态计算得出，避免状态同步不一致。

### 5. 会话一致性恢复与异常修复
- 系统在加载历史会话时，必须校验工具调用链的完整性。如果发现大模型发出了工具调用请求，但在文件截断前尚未写入执行结果，系统必须自动将对话历史回滚到该调用之前的最后一个一致性状态，以符合模型接口的强一致协议约束。
- 恢复加载时，如果检测到两次对话之间的时间间隔超过了设定的久闲阈值，系统必须自动在中间插入一条特定格式的用户提示消息，提醒模型上下文可能存在时效性问题。
- 恢复加载时，如果检测到总 Token 数量（包含会话历史、静态指令与记忆索引）超出了上下文限制，系统应在启动前自动触发一次就地历史压缩。

### 6. 过期会话清理
- 系统应定期在后台扫描并清理长期未被激活的过期会话，避免存储膨胀。

### 7. 主动记忆提炼与分类笔记
- 在每轮 Agent 循环自然结束（模型最终输出普通文本且无后续工具调用）后，系统应异步触发大模型对当前对话的提炼。
- 提炼出的记忆笔记应区分用户个人级与项目共享级，并分门别类存储（分类包括：用户偏好、纠正反馈、项目知识、参考资料）。
- 提炼与去重逻辑完全由模型自主判断。

### 8. 索引自动精简与注入
- 系统应维护一份长期记忆索引文件。项目指令与长期记忆索引仅在会话启动初始化时加载并注入 System Prompt，在当前运行会话期内保持静态，以最大化提示词缓存（Prompt Cache）的命中率。
- 索引文件必须严格限制行数与物理体积。在大模型更新索引内容时，若预测大小将超出限制，模型必须自主进行条目的合并或淘汰，程序本身不进行强行裁剪。

## 非功能要求
- **崩溃安全性**：使用追加式文件写入，在发生断电或被系统强杀时，已成功落盘的数据不受损坏。
- **轻量与易部署**：不引入向量数据库或本地嵌入式数据库，保持单 binary 执行文件的分发优势。
- **零漂移**：会话状态动态计算，彻底避免索引文件与实际会话文件内容发生漂移。

## 设计骨架
- **指令加载器**：读取并多级合并静态 Markdown 指令，支持递归解析引用标签，检测环路与沙箱安全。
- **会话持久化引擎**：管理会话的写（追加落盘）、读（动态解析与校验），并在启动时执行一致性截断、时间跨度提示插入与过期清理。
- **异步记忆收集器**：监听 Agent Loop 的结束事件，触发后台线程向模型请求生成笔记，并传入完整索引供模型自助合并或淘汰旧条目后写入磁盘。

## Out of Scope
- 本阶段不实现本地向量检索（RAG）。
- 本阶段不实现多端/团队之间的记忆网络云同步。
- 暂不引入本地物理嵌入模型（Embedding Model）。
