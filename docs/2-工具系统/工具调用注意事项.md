# 工具调用模块：API 格式、设计要点与常见问题

## 一、API 请求格式

### 1.1 Anthropic Messages API

```
POST https://api.anthropic.com/v1/messages
Headers:
  x-api-key: <key>
  anthropic-version: 2023-06-01
```

请求体（带 tools）：

```json
{
  "model": "claude-opus-4-7",
  "max_tokens": 32768,
  "stream": true,
  "system": "你是 bingtangCode，一个终端 AI 编程助手...",

  "tools": [
    {
      "name": "read_file",
      "description": "读取项目中的文件内容，返回带行号前缀的文本...",
      "input_schema": {
        "type": "object",
        "properties": {
          "filePath": {"type": "string", "description": "文件路径，相对于项目根目录"},
          "startLine": {"type": "integer", "description": "起始行号（1-based）"},
          "endLine": {"type": "integer", "description": "结束行号（1-based）"}
        },
        "required": ["filePath"]
      }
    }
  ],

  "messages": [
    {"role": "user", "content": "帮我读一下 pom.xml"}
  ]
}
```

关键点：Anthropic 用 `input_schema`（不是 `parameters`）。system prompt 放在顶层 `"system"` 字段，不在 messages 数组里。

### 1.2 OpenAI Chat Completions API

```
POST https://api.openai.com/v1/chat/completions
Headers:
  Authorization: Bearer <key>
```

请求体（带 tools）：

```json
{
  "model": "gpt-4o",
  "stream": true,

  "tools": [
    {
      "type": "function",
      "function": {
        "name": "read_file",
        "description": "读取项目中的文件内容...",
        "parameters": {
          "type": "object",
          "properties": {
            "filePath": {"type": "string", "description": "文件路径"}
          },
          "required": ["filePath"]
        }
      }
    }
  ],

  "messages": [
    {"role": "user", "content": "帮我读一下 pom.xml"}
  ]
}
```

关键点：OpenAI 用 `parameters`（不是 `input_schema`），且包在 `function` 对象里，外层加 `"type": "function"`。system prompt 是 messages 数组里一条 `role: "system"` 消息。

### 1.3 两种 API 格式对照

| | Anthropic | OpenAI |
|---|----------|--------|
| Schema 字段名 | `input_schema` | `parameters`（包在 `function` 内） |
| 工具包装 | 无，直接 `{name, description, input_schema}` | `{type:"function", function:{name, description, parameters}}` |
| system prompt | 顶层 `"system"` 字段 | messages 里 `role:"system"` |
| tool_call 的 type | `"tool_use"` | `"function"` |
| arguments 格式 | JSON 对象 `{"filePath":"a"}` | JSON 字符串 `"{\"filePath\":\"a\"}"` |
| tool_result 位置 | user 消息 content 数组里 | 独立 `role:"tool"` 消息，每条结果一条 |
| tool_result ID 字段 | `tool_use_id` | `tool_call_id` |

### 1.4 工具调用往返中的消息格式

模型调工具后，需要把执行结果回灌给 API。此时 messages 数组变成：

**Anthropic：**

```json
{
  "messages": [
    {"role": "user", "content": "帮我读一下 pom.xml"},

    {"role": "assistant", "content": "让我帮你看看", "tool_calls": [
      {"id": "toolu_01", "type": "tool_use", "name": "read_file", "input": {"filePath": "pom.xml"}}
    ]},

    {"role": "user", "content": [
      {"type": "tool_result", "tool_use_id": "toolu_01", "content": "  1  <project>...", "is_error": false}
    ]}
  ]
}
```

注意：`tool_result` 所在消息的 `content` 是**数组**（不是字符串），数组元素才是 `{type, tool_use_id, content, is_error}`。`is_error` 为 false 时可以不传，为 true 时必须传。

**OpenAI：**

```json
{
  "messages": [
    {"role": "user", "content": "帮我读一下 pom.xml"},

    {"role": "assistant", "content": "让我帮你看看", "tool_calls": [
      {"id": "call_01", "type": "function", "function": {"name": "read_file", "arguments": "{\"filePath\":\"pom.xml\"}"}}
    ]},

    {"role": "tool", "tool_call_id": "call_01", "content": "  1  <project>..."}
  ]
}
```

注意：OpenAI 的 tool_result 不是嵌在 user 消息里，而是独立的 `role:"tool"` 消息。每条结果一条消息。`arguments` 是 JSON 字符串，序列化时要用 `writeValueAsString`。

## 二、SSE 响应解析

### 2.1 Anthropic SSE 事件

**纯文本回复：**

```
event: message_start
data: {"type":"message_start","message":{"id":"msg_01","role":"assistant",...}}

event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"text","text":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"好的"}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"，pom.xml 里..."}}

event: message_delta
data: {"type":"message_delta","delta":{"stop_reason":"end_turn","stop_sequence":null}}

event: message_stop
data: {"type":"message_stop"}
```

处理逻辑：`content_block_delta`(type=text_delta) → `callback.onToken()`。`message_stop` → `callback.onComplete()`。

**工具调用回复（纯工具，无前置文本）：**

```
event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"tool_use","id":"toolu_01","name":"read_file"}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"input_json_delta","partial_json":"{\"filePath\":\""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"input_json_delta","partial_json":"pom.xml\"}"}}

event: message_delta
data: {"type":"message_delta","delta":{"stop_reason":"tool_use"}}

event: message_stop
data: {"type":"message_stop"}
```

**工具调用回复（先说话再调工具——text 和 tool_use 在同一流里共存）：**

```
event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"text","text":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"让我帮你读一下 pom.xml"}}

event: content_block_start
data: {"type":"content_block_start","index":1,"content_block":{"type":"tool_use","id":"toolu_01","name":"read_file"}}

event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"input_json_delta","partial_json":"{\"filePath\":\"pom.xml\"}"}}

event: message_delta
data: {"type":"message_delta","delta":{"stop_reason":"tool_use"}}

event: message_stop
data: {"type":"message_stop"}
```

index=0 是 text block（→ `onToken`），index=1 是 tool_use block（→ `onToolCall`）。两者互不冲突，在同一流里先后出现。最终组成一条 assistant 消息：`content="让我帮你读一下 pom.xml"` + `tool_calls=[read_file]`。

**工具调用回复（一次回复同时调多个工具）：**

```
event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"text","text":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"让我同时帮你查这两个文件"}}

event: content_block_start
data: {"type":"content_block_start","index":1,"content_block":{"type":"tool_use","id":"toolu_01","name":"read_file"}}

event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"input_json_delta","partial_json":"{\"filePath\":\"pom.xml\"}"}}

event: content_block_start
data: {"type":"content_block_start","index":2,"content_block":{"type":"tool_use","id":"toolu_02","name":"search_content"}}

event: content_block_delta
data: {"type":"content_block_delta","index":2,"delta":{"type":"input_json_delta","partial_json":"{\"query\":\"spring\",\"filePattern\":\"*.xml\"}"}}

event: message_delta
data: {"type":"message_delta","delta":{"stop_reason":"tool_use"}}

event: message_stop
data: {"type":"message_stop"}
```

index=0 是 text，index=1 是第一个工具，index=2 是第二个工具。`message_stop` 时按 index 升序依次回调：先 `onToolCall(read_file)` 再 `onToolCall(search_content)`。

处理逻辑：`content_block_start`(type=tool_use) → 按 index 分别记录 id + name 到各自的 `Map<Integer, ...>`。`content_block_delta`(type=input_json_delta) → 按 index 拼接到各自的 `StringBuilder`。`message_stop` → 遍历 Map 按 index 升序逐个 `callback.onToolCall()` → `callback.onComplete()`。

实现要点：**必须用 `Map<Integer, ...>` 按 index 分别存储，不能用单值变量**——否则第二个 tool_use 的 id/name/json 会覆盖第一个，导致第一个工具调用丢失。

### 2.2 OpenAI SSE 事件

**工具调用回复：**

JSON 流式 chunk 里 `choices[0].delta.tool_calls` 是一个数组，每个元素有 `index`（区分不同工具调用）、`id`、`function.name`（首次出现时）、`function.arguments`（JSON 字符串片段）。

```
chunk 1: delta.tool_calls = [
  {"index":0, "id":"call_01", "function":{"name":"read_file"}}
]

chunk 2: delta.tool_calls = [
  {"index":0, "function":{"arguments":"{\"filePath\":\""}}
]

chunk 3: delta.tool_calls = [
  {"index":0, "function":{"arguments":"pom.xml\"}"}}
]

chunk 4: finish_reason = "tool_calls"
```

处理逻辑：按 index 用 `Map<Integer, StringBuilder>` 分别累积 arguments 片段。`finish_reason=tool_calls` 时解析完整 JSON → `callback.onToolCall()` → `callback.onComplete()`。

### 2.3 思考内容（reasoning_content）

DeepSeek R1/V3 等推理模型会在 `delta` 里输出独立的 `reasoning_content` 字段，与 `content` 平行但互不包含。思考内容不进 `onToken` 回调——直接打印到终端，不写入对话历史。如果思考之后有正式回答，思考内容在终端上自动折叠为一行 `💭 思考 Xs` 摘要。

## 三、工具定义

### 3.1 Tool 接口

```java
public interface Tool {
    String getName();              // 唯一名称，如 "read_file"
    String getDescription();       // 一段详细描述，模型据此判断是否调用
    String getParametersSchema();  // JSON Schema 字符串，定义参数约束
    ToolResult execute(Map<String, Object> params);  // 执行工具
}
```

四个方法分两类：前三个是**元信息**——各 Provider 在构造 API 请求时把 name/description/schema 填入 tools 数组，告诉 LLM 有什么工具可用。第四个是**执行**——由 ToolExecutor 在模型发起调用时触发。

schema 放在 Tool 实现里返回纯字符串，不做 API 格式适配。Anthropic 需要 `input_schema`、OpenAI 需要 `parameters`——这个差异由各 Provider 自己处理，不污染 Tool 层。

### 3.2 ToolCall 和 ToolResult

```
ToolCall  = 模型说"我要调这个工具"
ToolResult = 工具说"这是执行结果"
```

一次工具调用由四者组成：

```
Tool.getId/name/params    →  SSE 解析得到的 ToolCall
Tool.execute(params)      →  返回 ToolResult
ToolResult.toolCallId     →  回链到原始 ToolCall.id
ToolResult.isError        →  模型据此判断是否要重试
```

## 四、常见问题

### Q1: 一次回复能调用多个工具吗？

能。SSE 流里每个 tool_use 块有独立 index：

```
index=0: content_block_start(type=tool_use, id="call_01", name="read_file")
index=0: content_block_delta(input_json_delta, partial_json=...)
index=1: content_block_start(type=tool_use, id="call_02", name="search_content")
index=1: content_block_delta(input_json_delta, partial_json=...)
```

用 `Map<Integer, ...>` 按 index 各自累积，不要用单值变量——第二个 index 会覆盖第一个。流结束时按 index 升序依次 `onToolCall()`。

DialogueManager 在 `onComplete` 时统一处理 `pendingToolCalls` 列表——遍历执行每个工具，结果收集到一起，打包回同一条 `USER + toolResults` 消息。

### Q2: 工具调用和文本输出能共存吗？

能。模型可以在调工具之前先解释意图。同一个 SSE 流里，各 content_block 按 index 独立运行：

```
index=0, type=text:       "让我帮你找一下配置文件"    → onToken()
index=1, type=tool_use:   read_file(filePath="...")    → onToolCall()
```

两者不是互斥的——`onToken` 和 `onToolCall` 先后触发，`onComplete` 时一起处理：

```java
history.add(ASSISTANT,
    "让我帮你找一下配置文件",   // content ← onToken 收集的文本
    [read_file],                // toolCalls ← onToolCall 收集的调用
    []                          // toolResults（尚未执行）
);
```

### Q3: 能多轮调用工具吗？

当前只支持**一轮**。`alreadyCalledTool` 控制：

```
第一轮 streamChat：
  → 模型返回 tool_call
  → 执行工具 → 结果回灌 history
  → 第二轮 streamChat (alreadyCalledTool=true)
      → 模型返回纯文本 → 正常结束 ✓
      → 模型又返回 tool_call → 丢弃调用，直接结束 ✗
```

如果需要多轮（如 `find_files` → `read_file` → 分析，需要两轮工具调用），把 `alreadyCalledTool` 布尔值改成 `int toolRound` 计数器，设上限 5 轮即可。

多轮实现时的关键约束：每轮 assistant+tool_calls 后必须紧跟 user+tool_results，否则 API 返回 400。

### Q4: 工具执行超时了怎么办？

ToolExecutor 用独立线程 + `Future.get(30s)` 实现超时控制。超时时 `future.cancel(true)` 强制中断工具线程，返回 `ToolResult(isError=true, content="工具执行超时（30秒）")`。模型收到 isError 后通常会自行调整——比如换个参数重试，或者告诉用户执行时间过长。

为什么必须开独立线程？因为 provider 线程是单线程 executor——如果在这个线程上直接调 `tool.execute()` 遇到死循环，整个程序的 SSE 响应能力就永远丢失了。

### Q5: 工具返回了 isError=true，模型会怎么处理？

模型会像人一样读错误信息、理解原因、调整策略。用一个 edit_file 的例子说明全流程。

用户说："把 README.md 里的 'install' 改成 'installation'"

**第一轮——模型发起工具调用：**

```
模型调用 edit_file(filePath="README.md", oldString="install", newString="installation")
```

**工具执行——发现 oldString 出现多次：**

README.md 里 "install" 这个词出现了 5 次（"To install..."、"run the install script"、"after install" ...）。EditFileTool 的逻辑是先统计 oldString 出现次数：

```java
int matchCount = 5; // "install" 出现了 5 次
return new ToolResult(null, "匹配到 5 处，请缩小范围", true);
```

**结果回灌——ToolResult 变成 API 消息喂给模型：**

```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01",
      "content": "匹配到 5 处，请缩小范围",
      "is_error": true
    }
  ]
}
```

**第二轮——模型读到错误，自己调整：**

模型看到 `is_error: true`，读到内容 "匹配到 5 处，请缩小范围"。它理解了——"install" 太泛了，需要更具体的上下文。于是它重新发起工具调用：

```
模型调用 edit_file(filePath="README.md",
    oldString="To install bingtangCode, run: npm install",
    newString="To install bingtangCode, run: npm installation")
```

这次 oldString 足够长、足够独特，只匹配 1 处 → 替换成功 → 返回 `isError=false`。

**对比：如果抛异常会怎样？**

```java
throw new RuntimeException("匹配到 5 处");
```

模型永远收不到这条信息——异常在 ToolExecutor 层被捕获，转成笼统的：

```json
{
  "content": "执行失败: 匹配到 5 处",
  "is_error": true
}
```

模型只知道"失败了"，不知道为什么，无法自我纠正。而精心写的 `"匹配到 5 处，请缩小范围"` 既告诉它问题（多处于匹配）又告诉它解法（缩小范围）。

**所有工具的错误信息都遵循这个原则——告诉模型出了什么事 + 暗示怎么修正：**

| 工具 | 错误场景 | 返回给模型的信息 |
|------|---------|----------------|
| edit_file | oldString 不存在 | `"未找到匹配文本"` |
| edit_file | oldString 出现多次 | `"匹配到 3 处，请缩小范围"` |
| read_file | 文件不存在 | `"文件不存在: pom.xml"` |
| read_file | 路径越界 | `"安全限制: 禁止访问项目目录以外的文件"` |
| execute_command | 用户取消 | `"用户取消了命令执行"` |
| ToolExecutor | 30s 超时 | `"工具执行超时（30秒）"` |

模型读到每一条都能做出合理反应——换路径、缩小匹配范围、确认用户意图、或者告诉用户超时了。

### Q6: 为什么 tool_calls 和 tool_results 要放在不同的 Message 里？

API 要求。Anthropic 的设计中，assistant 消息包含 tool_calls（模型发起的），user 消息包含 tool_results（外部返回的）。模型是"被动的"——它只能请求工具，不能自己执行。工具结果是"外部输入"，所以放在 user 消息里，模拟用户把工具结果告诉模型的场景。

### Q7: alreadyCalledTool 拒绝时为什么不能加 assistant+tool_calls 到 history？

API 要求 assistant+tool_calls 后面必须紧跟 tool 消息（OpenAI）或 user+tool_results（Anthropic）。如果只加了 assistant+tool_calls 但不执行工具（没有 tool_results 跟上），下一轮 API 请求就会拿到 400。

所以拒绝的正确做法是：**不记录这一轮的 tool_calls，直接加一句 user 提示让模型重试**。

### Q8: buildApiMessages 合并相邻同角色消息时要注意什么？

合并 content 文本之外，`toolCalls` 和 `toolResults` 列表也必须合并，否则会丢数据：

```java
new Message(prev.role(),
    prev.content() + "\n" + msg.content(),
    merge(prev.toolCalls(), msg.toolCalls()),       // 必须
    merge(prev.toolResults(), msg.toolResults()));   // 必须
```

### Q9: 搜索文件内容时为什么整个搜索会中断？

`Files.readAllLines()` 读到二进制文件（如 `.DS_Store`）会抛 `MalformedInputException`。如果外层 catch 直接返回 `isError=true`，整个搜索就停了。修复方式是单文件读失败时 `continue` 跳过，不影响其他文件的遍历。

### Q10: 工具路径安全怎么保证？

所有文件工具（read/write/edit）构造函数接收 `Path projectRoot`，execute 时做两步校验：

```java
Path resolved = projectRoot.resolve(filePath).normalize();
if (!resolved.startsWith(projectRoot)) {
    return ToolResult(isError=true, "禁止越界访问");
}
```

`normalize()` 把 `../../etc/passwd` 消解为实际路径，`startsWith()` 检查是否还在项目根下面。两个检查缺一不可。
