# 工具系统：调用链与设计动机

## 一句话总结

整个工具系统做了一件事——**把"模型想调工具"翻译成"执行代码"再翻译成"模型看得懂的结果"**。拆成多个类的唯一原因：Anthropic 和 OpenAI 的 API 格式不同、工具执行需要超时控制、模型可能会再调工具需要防无限循环。每一层只管一件事。

---

## 完整调用链（一次工具调用往返）

用户输入 `"帮我读一下 pom.xml"`，模型调用 `read_file` 工具。

### 第一步：发起请求

```
SessionManager（主线程）
  │  dialogue.addUserMessage("帮我读一下 pom.xml")
  │  dialogue.streamResponse(provider, callback)
  │  latch.await()  ← 主线程等待
  │
  ▼
DialogueManager.doStream(provider, callback, alreadyCalledTool=false)
  │
  │  创建 wrapper（StreamCallback 匿名类）拦截四个回调:
  │    onToken      → sb.append() + callback.onToken()  转发打印
  │    onToolCall   → pendingToolCalls.add()            静默收集
  │    onComplete   → 检查 pendingToolCalls，决定走文本还是工具流程
  │    onError      → 透传
  │
  │  调用 provider.streamChat(apiMessages, tools, wrapper)
  │  streamChat 提交任务到 provider 单线程 executor 后立即返回
  │  doStream 也返回——剩下的活全在 provider 线程异步跑
  ▼
```

### 第二步：组装 API 请求体

provider 线程（anthropic-stream-reader）开始跑 `doStreamChat()`。

`buildRequestBody()` 把内部数据转成 API 格式。以 Anthropic 为例：

```
内部 Message:
  [
    Message(SYSTEM, "你是 bingtangCode..."),
    Message(USER,   "帮我读一下 pom.xml")
  ]

内部 Tool（ToolRegistry.getAll()）:
  Tool(name="read_file", description="读取项目中的文件...", parametersSchema="{type:object,...}")
  Tool(name="write_file", ...)
  ...共 6 个

            ↓ buildRequestBody() 转换 ↓

API 请求体:
  POST /v1/messages
  {
    "model": "claude-opus-4-7",
    "max_tokens": 32768,
    "stream": true,
    "system": "你是 bingtangCode...",
    "tools": [
      {
        "name": "read_file",
        "description": "读取项目中的文件内容...",
        "input_schema": {"type":"object","properties":{"filePath":{"type":"string"},...}}
      },
      { "name": "write_file", ... },
      ...
    ],
    "messages": [
      {"role": "user", "content": "帮我读一下 pom.xml"}
    ]
  }
```

关键转换：
- SYSTEM 消息抽出来放顶层 `"system"` 字段（Anthropic 特有）
- Tool 的 `getParametersSchema()` 返回 JSON 字符串，反序列化后填入 `input_schema`（Anthropic 叫 input_schema，OpenAI 叫 parameters）
- 纯文本 USER 消息直接 `{"role":"user","content":"..."}`

### 第三步：SSE 解析 → 回调

provider 线程收到 SSE 流，`parseSSEStream()` 逐行解析。

**纯文本回复**（模型直接回答，不调工具）：

```
SSE 流:                                 代码回调:

content_block_start(type=text)          （忽略）
content_block_delta(text_delta="好")    → onToken("好")  → 终端打印 "好"
content_block_delta(text_delta="的")    → onToken("的")  → 终端打印 "的"
message_delta(stop_reason=end_turn)     （忽略）
message_stop                            → onComplete()
                                            → pendingToolCalls 为空
                                            → history.add(ASSISTANT, "好的，pom.xml是...")
                                            → callback.onComplete()
                                            → latch.countDown() → 主线程醒来
```

**工具调用回复**（模型要调 read_file）：

```
SSE 流:                                 代码回调:

content_block_start(index=0, text)      （忽略，先有一个空文本块）
content_block_start(index=1,
  type=tool_use,                        → 记录 currentToolUseId="toolu_01"
  id="toolu_01",                          记录 currentToolName="read_file"
  name="read_file")                       清空 partialJson

content_block_delta(index=1,
  type=input_json_delta,               → partialJson = "{\"filePath\":\""
  partial_json="{\"filePath\":\"")

content_block_delta(index=1,
  type=input_json_delta,               → partialJson = "{\"filePath\":\"pom.xml\"}"
  partial_json="pom.xml\"}")

message_delta(stop_reason=tool_use)     → stopReasonToolUse = true

message_stop                            → JSON 完整，解析为 Map
                                        → onToolCall(ToolCall("toolu_01","read_file",{filePath:"pom.xml"}))
                                              → pendingToolCalls.add(toolCall)
                                        → onComplete()
                                              → pendingToolCalls 非空
                                              → handleToolCalls()
```

注意：整个 SSE 解析过程中，`content_block_delta` 的文本片段由 index=0 的 text block 发送 → onToken。`input_json_delta` 片段由 index=1 的 tool_use block 发送 → 内部拼接。两个 block 独立推进，子同一个流里先后出现。

### 第四步：工具执行 + 结果回灌

`handleToolCalls()` 在 provider 线程上被调用（它是 onComplete 里触发的）。

```
handleToolCalls(provider, callback, textPrefix, toolCalls, alreadyCalledTool=false)
  │
  ├─ history.add(ASSISTANT, "让我帮你读一下", [read_file], [])
  │   内部: Message(ASSISTANT, content="让我帮你读一下",
  │                  toolCalls=[ToolCall("toolu_01","read_file",{filePath:"pom.xml"})],
  │                  toolResults=[])
  │
  │    → buildApiMessage 转译:
  │      {"role":"assistant","content":"让我帮你读一下",
  │       "tool_calls":[{"id":"toolu_01","type":"tool_use","name":"read_file","input":{"filePath":"pom.xml"}}]}
  │
  ├─ ToolRegistry.get("read_file") → ReadFileTool 实例
  │
  ├─ ToolExecutor.execute(readFileTool, "toolu_01", {filePath:"pom.xml"})
  │     │
  │     ├─ 提交到 tool-executor 线程
  │     ├─ provider 线程阻塞在 Future.get(30s)
  │     ├─ tool-executor: readFileTool.execute(params) → 读文件 → ToolResult(null,"  1  <project>...",false)
  │     └─ provider 线程拿到结果，注入 toolCallId:
  │        ToolResult("toolu_01","  1  <project>...",false)
  │
  ├─ history.add(USER, "", [], [toolResult])
  │   内部: Message(USER, content="", toolCalls=[], toolResults=[ToolResult("toolu_01","...",false)])
  │
  │    → buildApiMessage 转译:
  │      {"role":"user","content":[{"type":"tool_result","tool_use_id":"toolu_01",
  │                                 "content":"  1  <project>...","is_error":false}]}
  │   注意: user+toolResults 的 content 是数组，不是字符串
  │
  └─ doStream(provider, callback, alreadyCalledTool=true)  ← 递归发起第二轮
```

### 第五步：第二轮 API 调用

```
doStream(alreadyCalledTool=true)
  │
  │  此时 history = [
  │    SYSTEM:   "你是 bingtangCode...",
  │    USER:     "帮我读一下 pom.xml",
  │    ASSISTANT: {toolCalls: [read_file]},       ← 模型说的
  │    USER:     {toolResults: [文件内容]},        ← 代码执行的
  │  ]
  │
  │  buildApiMessages() → 清洗后发给 API:
  │    messages: [
  │      {"role":"user","content":"帮我读一下 pom.xml"},
  │      {"role":"assistant","content":"","tool_calls":[...]},
  │      {"role":"user","content":[{"type":"tool_result",...}]}   ← 模型看到这行
  │    ]
  │  provider.streamChat(...)
  ▼

SSE 流（第二轮）:
  content_block_delta(text_delta="pom.xml")     → onToken("pom.xml")
  content_block_delta(text_delta="里定义了")     → onToken("里定义了")
  ...
  message_stop                                   → onComplete()
                                                      → pendingToolCalls 为空
                                                      → history.add(ASSISTANT, "pom.xml里定义了...")
                                                      → callback.onComplete()
                                                      → latch.countDown()
                                                      → 主线程醒来，打印完成
```

如果第二轮模型又返回 tool_call：`alreadyCalledTool=true` → 不执行，注入 "本轮仅支持一次工具调用" → 再发一次。防止无限循环。

## 为什么需要这些类：逐层动机

### Tool / ToolCall / ToolResult（数据模型）

**为什么不是直接传 JSONObject？** 三者的语义完全不同——ToolCall 是"模型说要做什么"，ToolResult 是"做完了什么结果"。拆成独立类型让类型系统帮你检查——ToolRegistry 只收 Tool，DialogueManager 组装 ToolCall+ToolResult，不会把结果当成调用传给执行器。

**为什么 ToolResult.isError 是布尔不是异常？** "文件不存在"不是系统异常——是工具的正常输出。isError=true 让模型知道要调整参数重试，而不是程序崩溃。

### ToolRegistry（注册中心）

**为什么需要注册中心？** `register()` 的同名检查在启动时就报错，而不是等模型调用时才发现两个工具冲突。`get(name)` 让 DialogueManager 不知道具体工具类型——只管 "拿名字找到工具然后 execute"，加新工具不用改它。

### ToolExecutor（超时执行器）

**为什么开独立线程？** 工具代码不受信任——可能是模型生成的 shell 命令，可能 `sleep 999`。开独立线程后用 `Future.get(30s)` 阻塞等待——超时则 `future.cancel(true)` 强制中断。如果直接调 `tool.execute()`，provider 线程永远卡死，整个 SSE 流再也不会恢复。

```
provider 线程                         tool-executor 线程
     │                                      │
     ├─ Future.get(30s) ──阻塞──→          ├─ tool.execute(params)
     │                                      │    sleep 999...
     ├─ 30秒到, TimeoutException            │
     ├─ future.cancel(true) ──中断──→      ├─ InterruptedException, 线程停止
     └─ 返回 isError=true
```

### StreamCallback.onToolCall()

**为什么用回调不是返回值？** `streamChat()` 是异步的——提交到 provider 线程后立刻返回。SessionManager 已经用 latch.await() 在等了，不能再用同步返回值。回调是异步线程和主逻辑之间唯一的事件通道。

### LLMProvider 接口 + 两个实现

**为什么不在 Tool 层做 API 格式转换？** Anthropic 用 `input_schema`，OpenAI 用 `parameters` 且包在 `function` 里。如果让 Tool 接口知道这两种格式，每加一个新 API 就要改 Tool 接口。让 Tool 只返回纯 JSON Schema 字符串，各 Provider 自己翻译——Provider 数量少（2个），格式差异是 Provider 自己的事。

具体差异所在：

| | Anthropic | OpenAI |
|---|----------|--------|
| 工具定义 | `{name, description, input_schema: {...}}` | `{type:"function", function:{name, description, parameters: {...}}}` |
| tool_calls 的 type | `"tool_use"` | `"function"` |
| arguments 的格式 | JSON 对象 `{"filePath":"pom.xml"}` | JSON 字符串 `"{\"filePath\":\"pom.xml\"}"` |
| tool_result 位置 | user 消息 content 数组里 | 独立 `role:"tool"` 消息 |
| system prompt | 顶层 `"system"` 字段 | messages 里 `role:"system"` 消息 |

SSE 事件结构也不同——Anthropic 用 content_block_start/delta 事件分块，OpenAI 用 delta.tool_calls 数组按 index 累积。两者在各自的 parseSSEStream 里处理，外部无感知。

### DialogueManager（编排）

**为什么 doStream 用递归？** 工具执行完不是终点——模型需要看到结果才能生成回复。第二次调用和第一次逻辑完全相同（请求→流式→回调），递归复用同一套 callback 逻辑，不需要写两套几乎一样的代码。

**alreadyCalledTool 为什么是递归参数而不是全局变量？** 每一轮调用独立——第一轮 false（允许执行工具），第二轮 true（拒绝再调用）。用参数明确传递，比全局状态更清晰。

## 线程协作

```
主线程                    provider 单线程               tool-executor 线程
  │                           │                           │
  ├─ readLine() 等待输入       │                           │
  ├─ 拿到输入                  │                           │
  ├─ streamResponse()         │                           │
  ├─ latch.await() 阻塞       │                           │
  │                           ├─ SSE 解析...              │
  │                           ├─ onToken() → 打印         │
  │                           ├─ onToolCall() → 收集      │
  │                           ├─ onComplete()             │
  │                           ├─ ToolExecutor.execute()   │
  │                           │   └─ Future.get() 阻塞    ├─ tool.execute(params)
  │                           │                           │   读取文件/执行命令
  │                           │                           └─ 返回 ToolResult
  │                           ├─ 拿到结果                  │
  │                           ├─ doStream() 第二轮        │
  │                           ├─ SSE 解析...              │
  │                           ├─ onToken() → 打印         │
  │                           ├─ onComplete()             │
  │                           │   └─ latch.countDown()    │
  ├─ 醒来，回到循环顶部        │                           │
```

三类线程各司其职：主线程管 UI 循环，provider 线程管 SSE 通信和编排，tool-executor 线程管超时执行。
