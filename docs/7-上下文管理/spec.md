# 上下文管理（Context Management）设计规约

## 1. 背景与目标
随着 `bingtangCode` 对话的累积，尤其是多次或大量的工具执行结果（如读取大文件、大范围搜索等），会导致上下文长度急剧膨胀，最终导致 LLM 的上下文窗口溢出而导致服务瘫痪。
为了在有限的 Token 预算下支持长时间干活，本项目需要引入两层上下文压缩策略：
1. **轻量预防（单条消息大小管理）**：在工具结果产生时，如果过大则落盘存储，只在对话历史中保留预览和路径，预防单次膨胀。
2. **重量兜底（累积历史长度管理）**：当对话总体大小逼近上下文上限时，通过 LLM 自动生成结构化摘要，移去大部分历史消息，保留近期原文。

## 2. 目标用户
需要使用 `bingtangCode` 终端进行长时间、多轮、重度工具调用（如频繁读写、运行命令）的开发者。

## 3. 能力清单
- **大工具结果落盘预防**：自动拦截超长的工具返回结果，将其存盘。在对话历史的 `ToolResult` 中仅保留文件预览、原文件路径、原始大小 and 指示模型重新读取的提示。
- **单条消息合计超限裁剪**：当单条消息内多个工具结果累计超限时，优先将字节数较大的工具结果依次落盘，直到剩余结果的大小在安全线内。
- **自动重量兜底摘要**：在每次向 API 发送请求前，评估当前上下文估算大小，若逼近有效窗口上限，则移去老旧消息，用 LLM 生成一份包含九个固定板块的结构化摘要，同时在此摘要内原文保留所有被压缩的用户原始消息。
- **草稿剥离与工具禁用**：调用 LLM 生成摘要时，明确禁用所有工具，并强制模型先在特定标签内撰写分析草稿，生成后程序自动将草稿部分剥离丢弃。
- **边界提示追加**：摘要生成后，在生成的摘要消息中追加边界提示，指示模型不要盲目脑补代码，需要文件原文时通过特定工具重新读取。
- **手动指令触发压缩**：允许用户在终端输入特定指令，强行触发一次重量兜底压缩。
- **三击失败熔断**：如果摘要生成连续失败多次，程序必须自动熔断（停止自动触发），防止死循环，并保障对话能继续进行。

## 4. 非功能要求
- **Token 近似估算**：不引入繁重的 Tokenizer 依赖，通过上一次真实的 API 输入与输出 Token 消耗作为基准，跳过最近一次已计入真实输出的助理消息，并对后续新产生的内容进行基于字符数的轻量级估算。
- **窗口扣减与安全余量公式**：
  - $\text{有效窗口} = \text{物理上下文窗口} - 20,000$（固定预留给摘要输出）
  - $\text{自动触发阈值} = \text{有效窗口} - 13,000$
  - $\text{手动触发阈值} = \text{有效窗口} - 3,000$
  - 通过预留输出和安全空间，能够容忍估算偏差并避免临界抖动。

## 5. 设计骨架
- **SessionId 运行时生成**：启动时生成唯一的会话 ID，作为工具结果落盘的子目录路径。
- **ContextManager（上下文管理器）**：核心协调者。计算当前上下文大小，判断是否需要预防落盘或兜底摘要。
- **ContextCompressor（LLM摘要执行器）**：负责构造专用的摘要 Prompt，调用 LLM 接口，剥离草稿标签，并将新摘要与边界提示拼接。
- **TUI/Command 路由**：在终端读取用户指令时，拦截手动压缩指令并调用 `ContextManager`。

## 6. Out of Scope（暂不做）
- **精确的 Tokenizer 计算**：本阶段不使用如 `jtokkit` 等精确的分词器，仅进行字符数换算估算。
- **机器学习自适应压缩**：不动态根据对话复杂度微调安全余量或保留原文的条数。
