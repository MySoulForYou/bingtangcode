## 一、上下文管理核心端到端流程：从对话累积到历史压缩

为了帮助你建立全局观，以下是 `bingtangCode` 从单次工具结果审计、自动预防落盘、Token 实时估算、直至触发重量兜底生成摘要并重组历史的完整、连贯生命周期：

### 阶段 1：大工具结果拦截与轻量预防（单条消息大小管理）

1. **工具批量执行完毕**：
   * 在运行时对话的每一轮，`DialogueManager` 委托 `ToolExecutor` 执行 LLM 请求的工具（如 `read_file`、`grep_search` 等），并获得 `List<ToolResult>` 结果集。
2. **单项超限落盘检测**：
   * `DialogueManager` 调用 `preventOverrunAndSave(results)` 对结果集进行安全审计。
   * 针对结果集中的每一个 `ToolResult`，若其 `content` 转换为 UTF-8 字节的长度超过 **50,000 字节**（约 50KB）：
     * **创建落盘路径**：在项目根目录下的 `.bingtangcode/sessions/<sessionId>/tool-results/` 中为当前工具调用生成唯一文件。
     * **物理文件写入**：使用 `Files.writeString` 将完整无截断的工具结果写入该磁盘文件。
     * **生成头部预览**：提取工具结果的前 20 行或前 2048 字节（取较短者），由于中文等多字节字符直接按字节截断会产生破损乱码，程序通过遍历 code point 并累加 UTF-8 字节大小进行安全截断。
     * **内存结果替换**：将内存中的 `ToolResult` 内容替换为四部分预览格式（包含原始大小、头部预览内容、落盘绝对路径、提示模型使用 ReadFile 重新读取的边界说明）。该结果标记为 `isSaved = true`。
3. **单条消息合计超限裁剪**：
   * 审计完单项后，程序计算当前消息中所有**尚未落盘**的工具结果的累计大小。
   * 若累计大小超过 **200,000 字节**（约 200KB）：
     * 对尚未落盘的工具结果按字节数**从大到小进行降序排序**。
     * 依次对最大的工具结果执行落盘和预览替换操作。
     * 每替换一个结果，累计大小扣减对应项的字节数，直至剩余未落盘结果的合计字节数降至 **200,000 字节以下**。
   * 经过两层预防处理后的 `processedResults` 被组装成一条 `USER` 消息追加到 `history` 对话历史中。

---

### 阶段 2：估算与重量兜底压缩（累积历史长度管理）

4. **API 请求前的 Token 估算**：
   * 在向 API 发送下一轮对话请求前，`DialogueManager` 必须评估当前的上下文大小。
   * **增量估算法**：利用上一次真实的 API input tokens 消耗（由 `StreamCallback.onUsage` 锚定并更新的 `lastApiInputTokens` 和 `lastApiHistorySize`）作为基础。
   * **增量换算**：若 `history` 的 size 大于上一次的大小，则对增量部分的 messages 中的 text、toolCalls（JSON 化）以及 toolResults 按字符数进行累加，并除以常数 `3.5`（`ESTIMATE_CHARS_PER_TOKEN`）作为估算 Token，与 `lastApiInputTokens` 相加，得到当前上下文的估算 Token 数。
5. **触发兜底压缩判定**：
   * **自动触发**：检查 `autoCompressEnabled` 状态。若为 `true` 且当前估算 Token $\ge \text{context\_window} - 20,000 - 13,000$。其中 20,000 为摘要输出预留空间，13,000 为自动安全余量。
   * **手动触发**：用户在终端输入 `/compact`，若估算 Token $\ge \text{context\_window} - 20,000 - 3,000$（3,000 为手动安全余量，或无条件强行压缩）时，会调用 `AgentLoop.manualCompress()` 触发。
6. **双下界划分保留区间**：
   * 触发压缩后，[DialogueManager](file:///Users/laq/Documents/bingtangcode/src/main/java/com/bingtangcode/core/DialogueManager.java) 从 history 尾部（最新消息）向前遍历累加消息条数和估算 Token。
   * 当**“累计估算 Token $\ge$ 10,000” 并且 “累计消息条数 $\ge$ 5”** 两个下界都满足时停止。这部分最新消息作为“**近期原文**”继续保留。
   * 排除掉顶部的 `System Prompt`（索引 0 消息），剩下的早期消息均被划为“**待压缩历史**”。
7. **调用专用 LLM 生成结构化摘要**：
   * 将“待压缩历史”格式化为文本，并注入到 `ContextCompressor.generateSummary()` 中。
   * 构造专属的 `SYSTEM_PROMPT`（明确禁用工具、要求模型必须在 `<draft>...</draft>` 标签中写草稿、限定输出九个 Markdown 二级标题）。
   * 调用 `LLMProvider` 执行同步阻塞的流式请求。
8. **草稿剥离与边界提示拼接**：
   * 收到 LLM 完整输出后，程序用 DOTALL 正则表达式 `(?s)<draft>.*?</draft>` 将草稿部分彻底剥离丢弃。
   * 在剩余的正式摘要后拼接**边界提示消息**。
9. **重组历史与重置锚定**：
   * 将“待压缩历史”从原 history 中删除。
   * 将【正式摘要 + 边界提示】组装为一条新的 `USER` 角色消息，插入在顶层 `System Prompt` 之后。
   * 随后追加“近期原文”部分。
   * 重置 `lastApiInputTokens = 0` 和 `lastApiHistorySize = 0`，使下一次估算自动重新以字符数为基准计算。
10. **多轮重试与熔断**：
    * 若 LLM 摘要请求发生异常/超时，累计失败次数。
    * 若连续失败达到 3 次，将 `autoCompressEnabled` 设为 `false` 触发自动熔断，在控制台打印警告，但允许本次及后续的 API 正常交互继续运行，防止陷入死循环。

## 二、上下文管理解决什么问题

### 之前的问题

- **大文件读取溢出**：当读取一个较大的项目源码文件（如 100KB），工具结果会全量回灌到对话历史。由于一次性注入的大量 Token，模型在下一轮请求时会立即因超出上下文限制（HTTP 400 Context Window Exceeded）而崩溃。
- **对话疲劳与费用暴涨**：随着交互轮数增多，早期已经解决的问题和早期的工具结果（比如旧的编译报错和搜索结果）依然作为历史源源不断地发送给 API。不仅白白消耗高昂的 Token 费用，还会因无关噪音过多导致模型表现退化、胡乱脑补。
- **猜测性补全**：模型会照着以前旧的工具结果（例如半截的文件内容）在后续对话中做猜测性补全，编写出很多语法残缺的代码。

### 现在的能力

| 能做 | 不能做 |
|------|--------|
| 配置驱动的上下文窗口定义 | 运行时根据模型复杂度自动调整安全余量 |
| 单个工具结果超限（50KB）落盘保护 | 对本地生成的摘要消息进行二次压缩（增量摘要） |
| 单条消息多工具累计超限（200KB）贪心落盘 | 精确分词计算（不依赖任何第三方分词器库） |
| 估算锚定（基准真实 token + 字符换算增量） | 在重量压缩时对顶层 System Prompt 实施修改 |
| 保留近期原文以维持即时记忆（ token $\ge 10\text{k}$ 且 消息 $\ge 5$） | 自动清理落盘的工具文件（落盘文件将作为会话历史保留） |
| LLM 摘要草稿剥离与工具禁用强制执行 | 跨会话的上下文压缩共享 |
| 手动指令强制压缩（`/compact`） | |
| 摘要连续失败 3 次自动熔断，防止死循环 | |
| 补发边界提示，指引模型使用 ReadFile 重新读取 | |

### 工具结果落盘的“主要工作区间”与“防爆定位”

工具落盘保护（由 `tool.result_limit` 与 `tool.result_total_limit` 协同控制）并非是一个高频触发的日常读写工具，而是一个系统级的**“安全气囊”**与**“熔断保险丝”**。

在真正高效的开发流程中，大模型作为“优等生”，会倾向于使用**“全局 Grep 搜索定位（SearchContent） -> 精准范围小段读取（read_file 限制 start/end 行号）”**的最高效模式工作，此时由于数据量极小（仅几百字节），**完全不会触发落盘保护**。然而，落盘机制在以下 10% 的“异常或高负载”工作区间内扮演着不可替代的防爆兜底角色：

1.  **防御大模型的“决策跑偏”或“盲目大范围通读”**：
    大模型在思考走偏，或者搜索无果时，可能会情急之下直接调用 `read_file` 强行读取一个 **3MB 的二进制 Shaded Jar 包** 或一个 **500KB 的超大日志配置文本**。
    *   **无落盘**：几百万字节的乱码/冗余文本会一次性强行灌满对话历史。下一轮 API 交互不仅会直接扣除高额 Token 费，还会因为上下文里堆满噪音而使模型直接“智商退化”。
    *   **有落盘**：系统在数据回灌前瞬间切断，将这几兆的原文冷存储到磁盘上，并在 Context 里只留下 20 行预览及路径提示。既保护了当前对话免受污染，又为模型在磁盘上留了一扇能通过行号“按需重读”的冷备窗口。

2.  **防御“高冗余编译/测试构建日志爆仓”**：
    模型在改完代码后通常会调用 `execute_command(command: "mvn test")` 跑单元测试。如果代码中引入了死循环，或者大量单元测试崩溃并打印出上千行、多达 **300KB 的 Exception 堆栈报错信息**：
    *   **无落盘**：模型会被超长日志直接撑爆上限。
    *   **有落盘**：因为累计结果超限，300KB 日志被强行拦截落盘。大模型通过预览里的前 20 行就已经能准确定位到第一个 NullPointerException 的报错位置，完美阻断了冗余控制台日志对 Context 的爆仓轰炸。

## 四、轻量级 Token 估算模型

在上下文管理中，我们并没有引入类似 `jtokkit` 的精确分词器依赖。因为：
1. **性能开销大**：每次请求前对几十万字的历史反复进行分词，会造成严重的 CPU 开销和卡顿。
2. **多语言混合复杂**：中英文、代码和特殊符号的分词器表现不一，纯依赖本地分词容易与云端 API 实际的计算产生不可预测的偏差。

因此，`bingtangCode` 设计了**“锚定真实数据 + 增量字符换算”**的轻量级估算模型：

### 1. 估算公式
$$T_{estimate} = T_{last\_api\_input} + T_{last\_api\_output} + \lceil \frac{C_{delta}}{3.5} \rceil$$

*   $T_{last\_api\_input}$：**上一次 API 请求返回的真实 input\_tokens**。它是在 `StreamCallback.onUsage` 中拿到并被锚定下来的，代表了直到上一轮请求时输入历史的精确 Token 数。
*   $T_{last\_api\_output}$：**上一次 API 请求返回的真实 output\_tokens**。它也是在 `StreamCallback.onUsage` 中被锚定，用于精确表示最近一条已完成的助理回复消息所消耗的真实 Token 数量。
*   $C_{delta}$：**增量字符数**。为了避免重复计算已经被 $T_{last\_api\_output}$ 精确代表的助理消息，增量字符计算会跳过索引为 `lastApiHistorySize` 的最近一则助理回复（即从索引 `lastApiHistorySize + 1` 开始），只累加此后新产生的 `Message` 的字符总和（包含新增的工具执行结果、用户新一轮的提问等）。
*   `3.5`（`ESTIMATE_CHARS_PER_TOKEN`）：**字符/Token 换算系数**。这是针对代码、中文、英文混合场景设计的统计平均值。即大约每 3.5 个字符算作 1 个 Token。

### 2. 为什么该模型能自纠偏？
因为一旦发生重量压缩或者发起了新的 API 请求并收到了回包，`lastApiInputTokens` 和 `lastApiOutputTokens` 就会在 `onUsage` 回调中被自动更新为 API 传回的**最精确的真实 Token 值**，同时 `C_{delta}` 清零。这意味着我们的估算误差**绝不会进行跨轮累积**，每一轮新的 API 响应都会对上一轮的估算误差进行一次“终极纠偏”。

## 六、重量兜底的滑窗保留与 LLM 摘要机制

### 1. 保留近期的双下界判定逻辑
当压缩发生时，我们不能把所有的历史消息全部摘要掉，否则大模型会失去最新的会话上文（比如它上一轮刚向用户提问，或者正在就某个具体的代码段进行微调）。
我们通过双下界逻辑从尾部向前数：
*   **估算 Token $\ge$ 10,000**：保证大模型拥有至少约 10,000 Token 的“即时记忆区”。
*   **消息条数 $\ge$ 5**：保证大模型至少能看到最近 5 条原汁原味的对话（通常包含两至三轮的问答与工具交互），维持基本的语境连贯。
*   **双下界必须同时满足**：只有当条数满足且 Token 够大时才停止，以保证最新记忆既不会因为消息条数过少而断裂，也不会因为消息太短而导致模型失去细节。

### 2. 禁用工具与草稿剥离的设计内幕
在调用 LLM 为早期历史生成摘要时，如果我们直接使用当前的 Provider 发送请求，模型可能会产生幻觉去调用现有的工具（比如它在早期历史中看到有 `ReadFile`，它在摘要时可能也想跑一下 `ReadFile`）。
*   **禁用工具**：`ContextCompressor` 在调用 `provider.streamChat` 时传入的 tools 是一个 `List.of()` 空列表。由于大模型拿到的“工具说明书”为空，它在写摘要时绝对无法使用任何工具。
*   **草稿隔离**：很多大模型在直接输出摘要时容易遗漏细节。因此在 SYSTEM 中，我们强迫模型使用 `<draft>` 标签先输出思考草稿（理顺思路、提取要点），随后输出正式的九大板块摘要。
*   由于草稿部分只是模型为了理清思路而生成的中间态，对于用户和下一轮对话而言是无意义的垃圾信息，程序在收到后，会利用正则表达式：
    ```java
    rawResponse.replaceAll("(?s)<draft>.*?</draft>", "").trim();
    ```
    将其彻底剔除。其中 `(?s)` 是 Java 正则的单行模式（`Pattern.DOTALL`），使句点符号 `.` 可以匹配包含换行符在内的任意字符，保证了多行草稿能被完美清除。

## 七、重要注意事项与设计细节

### 1. 输出预留（20,000 Token）的必要性
在计算压缩阈值时，我们加入了 `20,000` 的输出预留空间。因为：
如果模型上下文物理上限是 200K，而我们的输入历史已经达到了 199K。此时如果我们调用 LLM 生成摘要，由于输入占用了 199K，大模型在本次请求中最多只能说出 1K 的字（200K - 199K = 1K）。当摘要长度超过 1K 时，LLM 会因为超出窗口被强制截断，导致我们生成的摘要不完整甚至发生报错。
因此，必须提前扣除 20K 的“摘要输出预留”，确保大模型在被要求生成摘要时，依然有充足的吐字空间。

### 2. 自动触发与手动触发的余量区分
*   **自动触发安全余量（13,000 Token）**：自动触发时余量较大。因为这是在后台静默运行的，较大的安全空间可以容忍字符估算模型在极端长文本下的累积偏差，确保绝对不会在 API 边界线上翻车。
*   **手动触发安全余量（3,000 Token）**：手动触发是用户在终端输入 `/compact` 显式发起的。因为是用户主动要求，我们将余量收窄到 3,000 Token。如果估算大小在物理窗口的警戒线内，会立即对历史进行压缩，腾出空间。

### 3. 重新读取边界提示的作用
历史压缩后，大量的工具执行结果（例如读取到的文件原文）被移除了，只在 `USER` 摘要消息中留下了 `## 3 文件和代码段` 的概括信息。
如果下一轮对话中用户对模型说：“*现在修改刚才那个文件*”。模型如果仅凭摘要，可能会产生幻觉自己“脑补”文件内容进行修改，从而写出残缺代码。
因此，我们在摘要尾部拼接了醒目的边界提示消息（**重要提示：需要文件原文时，请使用 ReadFile 工具重新读取对应路径**）。大模型阅读到这个强硬的约束后，就会表现得非常规矩，在需要修改代码时，会先老老实实地调用一次 `read_file` 重新读取文件，再做修改。

### 4. 阈值的可配置性（测试与开发友好）
为了方便开发调试和不同场景下的极速功能验证，单项大工具结果落盘阈值（`toolResultLimit`）和单条消息累计超限阈值（`toolResultTotalLimit`）均已被提取为 `config.yaml` 配置文件中的配置参数：
*   `tool.result_limit`：默认值 `50000` 字节。
*   `tool.result_total_limit`：默认值 `200000` 字节。

在测试大工具结果落盘时，你可以临时在 `config.yaml` 中将它们分别设为非常小的值（如 `100` 和 `500`），无需制造超大文件就能非常直观、轻量地验证整个落盘拦截和预览流。
