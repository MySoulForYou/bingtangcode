# flow-guide.md · bingtangCode 命令系统

面向小白与开发者。读完本篇文档，你将全面掌握 `bingtangCode` 交互式命令系统的核心架构、12 个高频命令的具体实现、从用户输入到任务结束的多层运行交互流程，以及在面试中关于 Agent 架构设计与状态控制的深度考点。

## 一、本章节实现的功能清单

本章在 `bingtangCode` 终端交互助手中，全面设计并构建了一套**命令拦截、解析与分发系统（Slash Command System）**。主要实现的功能如下：

1. **命令行 `/` 前缀拦截与分流**：
   - 在用户输入的第一入口进行智能过滤。只要输入以 `/` 开头，系统便判定为控制命令，直接拦截并绕过大模型（LLM）对话主循环，零网络延迟响应。
   - 对空输入或纯 `/` 输入进行防呆拦截，避免将其作为普通文本发送给大模型，节省 Token 损耗。

2. **声明式命令集中注册与启动期防撞校验**：
   - 提供统一的命令注册表，管理命令的名称、别名、说明、类型、参数提示等元数据。
   - **启动阶段冲突检测（Fail-Fast）**：系统在初始化阶段自动扫描所有命令及别名，一旦发现命名冲突或别名碰撞，立即抛出 `IllegalStateException` 强行终止程序，杜绝带病运行。

3. **三种命令执行模式的分发（CommandType）**：
   - **LOCAL（纯本地只读命令）**：无需修改会话状态，只在本地读取并打印系统状态，例如 `/status`、`/memory`、`/permission`、`/session`、`/help` 等。
   - **LOCAL_UI（界面控制/会话操作命令）**：控制终端界面行为或切换会话，例如 `/plan`、`/do`、`/clear`、`/resume`、`/exit`。
   - **PROMPT（提示词模板命令）**：将预设提示词拼装后注入会话历史，并自动唤醒大模型对话轮次，例如 `/review`。

4. **界面控制抽象（UIController 接口）**：
   - 抽象出终端控制接口，将命令的执行逻辑与底层控制台的渲染细节（如 JLine3）彻底解耦，为自动化测试和未来图形界面扩展打下坚实基础。

5. **Tab 智能自动补全（Completer）**：
   - 挂载自定义 JLine 补全器，输入前缀（如 `/co`）按 Tab 键可瞬间匹配补全。
   - 过滤设置为隐藏（`hidden = true`）的调试命令，确保用户体验干净。
   - 支持多重候选联想，输入 `/p` 按 Tab 可在下方列出 `/plan`、`/permission` 菜单。

6. **交互式 TUI 选择列表（ActionListBox）**：
   - 在终端 Raw Mode（原始流模式）下工作，捕获用户键盘的上下方向键（或 `j`/`k`）、回车及 ESC 键。
   - 提供防闪烁的 ANSI 控制台动态重绘技术，提供类似于系统级菜单的会话恢复交互界面。

7. **12 个预置高频内置命令**：
   - `/exit` (或 `/quit`, `/q`): 正常退出程序。
   - `/plan` (或 `/p`): 切换至 Plan 只读权限模式，Prompt 状态栏高亮同步。
   - `/do` (或 `/d`): 切换至 Default 默认读写权限模式，Prompt 恢复默认。
   - `/compact` (或 `/c`): 手动触发会话历史压缩归档。
   - `/status` (或 `/s`): 按 key-value 对齐格式输出 Mode、Tokens、Tools、Memories、Model、Directory 六个系统指标。
   - `/memory` (或 `/m`): 逐行输出当前已加载的项目级与用户级 `.md` 记忆文件名（排除 `MEMORY.md` 索引）。
   - `/permission`: 仅打印当前的权限模式名称（如 `plan` / `default`）。
   - `/session`: 输出当前会话 ID 及 `.jsonl` 存档的规范化绝对路径。
   - `/review`: 拼装内置代码审查 prompt 并立即唤醒 AI 进行审查。
   - `/clear`: 关闭当前会话、重置内存消息与 Token 计数、清屏并重新生成新会话存档。
   - `/resume` (或 `/r`): 启动 ActionListBox 列表，通过键盘上下键选择并无缝加载历史会话。
   - `/help` (或 `/h`, `/?`): 按字母升序整齐打印命令帮助卡片。

## 二、功能实现讲解与技术细节

命令系统的整体运转围绕着**“拦截 $\rightarrow$ 解析 $\rightarrow$ 路由 $\rightarrow$ UI 呈现”**的闭环展开：

### 1. 输入拦截与大小写不敏感解析
- **位置**：`SessionManager.java` 的主循环。
- **解析机制**：
  - 当拿到用户输入 `input` 时，剔除首尾空白。若为空行直接跳过本轮循环。
  - 检测首字符是否为 `/`。若是，则使用正则表达式或首个空格位置将输入切分为命令名（Command Name）与参数字符串（Arguments）。
  - 例如输入 `/PLAN file.py --debug`，分流器切分出第一个空格前的内容，剔除开头的 `/` 并转换为全小写：`name = "plan"`，其余部分作为 `arguments = "file.py --debug"` 传入执行上下文 `CommandContext`。
  - 若输入仅为单独的 `/` 字符，为了防呆，系统会自动将其路由给 `/help` 逻辑以打印帮助文档。

### 2. 集中式命令注册与 Fail-Fast 冲突校验
- **类**：`CommandRegistry.java`
- **逻辑**：
  - 注册中心维护一个 `Map<String, Command> commands`，将命令名称的小写作为 Key 存入。
  - 同时维护 `Set<String> registeredNamesAndAliases` 记录所有已经注册的命令名 and 别名。
  - 在启动装配期（`Main.java` 中循环注册 Built-In 命令），每当注册一个新命令时：
    1. 校验其 `name.toLowerCase()` 是否已存在于 Set 中，若存在，说明发生重名碰撞；
    2. 遍历其 `aliases`，将别名转为小写并校验是否已存在于 Set 中，若存在，说明发生别名碰撞。
    3. 一旦碰撞，立即抛出 `IllegalStateException("命令名或别名冲突: ...")`，中断 JVM 启动。

### 3. JLine3 Tab 智能补全的挂载
- **类**：`TerminalIO.java` 内部的 `JLine Completer`
- **逻辑**：
  - 实现 JLine 的 `Completer` 接口中的 `complete(LineReader reader, ParsedLine line, List<Candidate> candidates)` 方法。
  - `ParsedLine` 提取用户当前已经输入的文本前缀（如以 `/` 开头）。
  - 调用 `CommandRegistry.complete(prefix)`。该方法首先过滤掉 `cmd.isHidden() == true` 的隐藏命令。
  - 然后仅匹配命令的主名称 `name` 是否以该前缀开头。若是，则将其拼装为带斜杠的完整形式（如 `/compact`）作为 `Candidate` 候选对象存入列表并返回给 JLine 渲染。这样做可以避免用户在输入 `/` 时，候选菜单中同时出现 `/c`（别名）和 `/compact`（全名）这类重复度极高的候选项。
  - JLine3 底层会自动判断：如果候选者仅有一个，直接填满命令行并上屏；若有多个，则在光标下方渲染一个可上下切换的选择菜单。

### 4. ActionListBox 的终端光标重绘与 Raw Mode 键盘监听
- **类**：`ActionListBox.java`
- **逻辑**：
  - **Raw Mode（原始流模式）**：在普通的终端输入中，程序必须等到用户按回车键才能拿到输入数据。`ActionListBox` 通过 JLine 开启终端的 Raw 模式，暂时关闭 canonical（标准）模式和终端回显（echo）。这样键盘上的每一次按键（包括上下方向键）都可以立刻以字节流的形式被程序捕获。
  - **防闪烁光标回滚（ANSI Escape Code）**：
    - `printBox` 首先在光标当前位置向下打印 `M` 行的选择菜单边框和选项内容（选中项应用反显和加粗 ANSI 控制符 `\033[7m\033[1m`）。
    - 当捕获到键盘的 `UP`（读取到 ANSI 序列 `\033[A`）或 `DOWN`（`\033[B`）时，计算最新的选中项索引。
    - **核心擦除逻辑**：调用 `clearBox(items.size())`。它向标准输出写入 ANSI 控制字符：`\033[N A`（将光标向上移动 N 行，N 为盒子打印的总行数）和 `\033[J`（擦除光标至屏幕末尾的全部字符）。
    - 擦除旧盒子后，立即调用 `printBox` 绘制带有新高亮位置的选择菜单。因为整个计算和重绘在几微秒内完成，用户在视觉上完全感受不到闪烁，获得了平滑的交互菜单体验。
    - 确认或取消后，再次调用 `clearBox` 擦除整个盒子，不污染用户控制台的历史行。

## 三、项目运行与多层交互流程（从输入到结束）

为了看清命令系统与底层 Agent 对话主循环、持久化层、工具层的交互方式，以下分别梳理**命令路由（本地）**与**大模型对话（普通输入及提示词命令）**两条主线的端到端执行流：

### 1. 本地命令运行流（以用户运行 `/plan` 或 `/clear` 为例）

```
 [用户输入 /plan]
       │
       ▼
 [TerminalIO] ─(Raw Input)─► [SessionManager (命令行分流器)]
                                      │
                                      ├─► 解析命令名: "plan", 参数: ""
                                      │
                                      ▼
                           [CommandRegistry] (检索 plan 命令元数据)
                                      │
                                      ▼ (匹配成功)
                           [CommandContext] (组装执行上下文，注入 UIController/Dialogue/Loop/Config)
                                      │
                                      ▼
                           [BuiltInCommands] (执行对应的 CommandHandler 回调)
                                      │
       ┌──────────────────────────────┴──────────────────────────────┐
       ▼ (如果是 /plan 命令)                                         ▼ (如果是 /clear 命令)
 [UIController.setPlanMode(true)]                             [DialogueManager.switchSession(...)]
  └─► 切换系统 PermissionMode                                  ├─► 清空内存消息历史
  └─► JLine Prompt 前缀高亮设为 [PLAN]                           ├─► 重新将静态 System Prompt 写入新文件
                                                               ├─► 重置 AgentLoop 运行状态计数器
                                                               └─► 调用 UIController.clearScreen() 清屏
```

### 2. 对话/提示词命令运行流（以普通消息或 `/review` 提示词注入为例）

```
 ┌──────────────────────┐     [用户输入 /review]
 │  用户输入普通对话文本  │            │
 └──────────┬───────────┘            ▼
            │                  [CommandRegistry] (匹配成功并调用 Handler)
            │                        │
            │                        ▼
            │                  [UIController.sendUserMessage(Prompt模板)]
            │                        │
            ▼                        ▼
      [SessionManager] ────────► [AgentLoop.run(userInput)] (开启一轮大模型迭代)
                                     │
                                     ├─► DialogueManager.addUserMessage(userInput) -> 写入 JSONL 存档 (WAL)
                                     │
                                     ▼ (迭代循环开始)
                               [LLMProvider.streamChat(...)] (大模型流式推理)
                                     │
                                     ├─► 若 LLM 输出普通文本 ──► TerminalIO (实时打字机打印) ──► 循环结束
                                     │
                                     ▼ (若 LLM 生成 Tool Calls 意图)
                               [ToolRegistry] (检测可用工具列表)
                                     │
                                     ▼
                               [PermissionGate] (进行权限安全网关校验)
                                     │
                                     ▼ (放行/拦截)
                               [ToolExecutor] (并行/单线程执行本地 Shell/File 等工具)
                                     │
                                     ▼
                               DialogueManager.addMessage(ToolResult) -> 再次 WAL 落盘
                                     │
                                     ▼ (重新流式聊天)
                                [LLMProvider] ──► 生成最终解答 ──► 退出 Agent 循环
                                     │
                                     ▼ (异步非阻塞触发)
                               [AutoMemoryCollector] (异步提炼更新 MEMORY.md 索引)
```

## 四、面试深度考点与技术问答 (Interview Q&A)

在面试中，围绕本章节中命令系统与 Agent 协作的核心架构，面试官通常会提出以下关于 **Agent 架构设计** 的深度考点：

### Q1: 为什么要在 Agent 的外层设计一个完全隔离的“命令系统（Command System）”拦截斜杠指令，而不是直接将所有输入（如 /clear, /status）投递给 Agent 并让大模型通过 Intent Classification（意图分类）或 Tool Calling 自动处理？
- **答**：
  主要基于以下四个核心考点进行权衡：
  1. **响应确定性（Deterministic Control）**：大模型的输出是概率性的。如果依靠大模型识别 `/clear` 意图，一旦发生幻觉，可能会误删会话或者未能成功清屏。而本地命令系统通过物理硬编码拦截，保证了 100% 的行为确定性。
  2. **冷启动与网络依赖（Offline Capability）**：如果网络中断，Agent 无法工作。但用户依然需要能本地运行 `/status` 查看状态、`/memory` 查看已加载记忆、或 `/exit` 优雅退出。独立命令系统不依赖任何 LLM 网络调用，能在 1ms 内瞬时执行。
  3. **运行开销与 Token 优化（Cost & Latency）**：如果每次输入清屏指令都要求助大模型，不仅会产生数百毫秒的网络延迟，还要平白支付数百 Token 的上下文和生成费用，极度不划算。
  4. **安全与特权操作（Privileged Actions）**：修改工作目录、关闭当前存档文件、强行改变模式等操作属于特权级别。如果向 Agent 开放这种系统级状态改变权限，会极大增加 **Prompt 注入攻击（Prompt Injection）** 的风险，必须在 LLM 处理环路之外做硬拦截。

### Q2: 系统的 `/plan` 和 `/do` 改变了 `PermissionMode`。从 Agent 执行管道的“纵深防御（Defense in Depth）”设计角度来看，这种状态切换是如何对底层工具层产生约束的？
- **答**：
  1. **非 LLM 自我约束**：在设计 Agent 安全性时，如果仅在 System Prompt 中写 `“你在 Plan 模式下不能修改文件”`，这属于“软约束”，很容易被用户通过精妙的 Prompt 注入来绕过。
  2. **网关硬拦截（Hard Gatekeeping）**：本章节的设计中，`/plan` 改变的是底层的 `PermissionMode`。当 Agent 迭代中 LLM 生成了 `tool_use` 想要修改文件时，[PermissionGate](file:///Users/laq/Documents/bingtangcode/src/main/java/com/bingtangcode/permission/PermissionGate.java) 会获取当前主线程的 `PermissionMode` 并比对工具的属性。如果是写操作工具且处于 `PLAN` 模式，会直接在网关层将该操作拦截，并返回给大模型一个拒绝结果（如 `"Permission denied: write tools are not allowed in PLAN mode"`）。这构成了一种底层的强限制，完全不以大模型自己的主观意志为转移。

### Q3: 内置命令 `/review` 属于 `PROMPT` 类型。它在触发时，为什么不直接在 Handler 内部发起一次 LLM Client 的点对点调用获取审查结果，而是大费周折地将其注入到 `UIController` 传回 `AgentLoop.run()`？
- **答**：
  这是为了**维护 Agent 上下文链条（Message Chain）的一致性**与**启用完整的 React-loop 工具执行链**：
  1. **防上下文断层**：如果在命令 Handler 内部直接点对点向 API 提问，生成的结果只会在控制台闪现一下，无法被后续的多轮对话历史记录（`DialogueManager` 中的 `history`）感知。下次发问时，大模型会遗忘刚刚做过 `/review` 这回事。
  2. **WAL 安全落盘**：通过 `UIController.sendUserMessage()` 触发，本质是将这一串审查指令作为正规的 `Role.USER` 消息记录到内存，并立即追加写落盘到当前的 `.jsonl` 存档中。
  3. **赋予 Agent 交互能力**：代码审查（`/review`）可能不是单次文本回答能解决的，Agent 可能需要运行 `read_file` 去看别的源文件，或者运行 `execute_command` 编译项目。如果直接点对点调用，Agent 就失去了使用工具的能力；而注入 `AgentLoop.run()` 能让审查流程无缝进入标准的 ReAct 轮次，大模型可以自由组合工具链最终得出深度审查报告。

### Q4: 当执行 `/clear` 时，系统不仅清空了内存的消息历史，还做了一次“关闭当前会话存档文件、重新开辟新 Session ID 并写入 System Prompt”的操作。这在 Agent 的持久化（Persistence）与崩溃恢复设计中扮演了什么角色？
- **答**：
  1. **防止冷启动状态漂移**：`bingtangCode` 采用了 Write-Ahead Logging（WAL）的设计思想，每次对话消息在核心节点完成后都会单行追加落盘到磁盘的 `.jsonl` 存档。如果 `/clear` 仅仅清空内存中的 `List<Message>`，那么一旦程序意外崩溃或重启，恢复加载机制（`SessionRecovery`）在扫描磁盘会话时，仍然会全量读入旧的 `.jsonl`，导致“清空”操作失效。
  2. **双重落盘一致性**：所以 `/clear` 的底层设计是一个**文件级的重置操作**。它在物理上关闭并锁定了当前的 `.jsonl` 句柄，通过随机数/时间戳产生一个全新的 `newSessionId` 并在磁盘上立刻创建一个干净的 `<newSessionId>.jsonl` 文件，随后将当前的 System Prompt 作为首行存入。这确保了在程序发生任何级别崩溃时，冷启动读取的磁盘状态与内存中的“已重置会话”绝对一致。

### Q5: 从 LLM API 的 Prompt Caching（提示词缓存）机制来看，为什么在命令系统中不应该提供“动态实时修改并更新 System Prompt”的本地命令？
- **答**：
  1. **缓存前缀匹配原理（Prefix Matching）**：包括 Anthropic（Claude）和 OpenAI（GPT）在内的 Prompt Caching 机制，都要求大模型请求的**提示词头部（前缀）必须完全保持不变**，才能命中云端的缓存，命中后可以节省高达 90% 的首字延迟（TTFT）和大量的 Token 计费。
  2. **高频失效隐患**：如果命令系统支持一个动态命令如 `/edit-prompt "你现在扮演Java专家..."` 去动态实时热修改 `DialogueManager` 里的 System Prompt，会导致从那一刻起，该会话头部发生改变，后续的所有对话历史的云端 Prompt Cache 瞬间全部 Miss。为了避免这种昂贵且高延迟的代价，我们把 System Prompt 设计为只在启动或开启全新会话（`/clear`）时静态组装，在运行中保持静止，从而最大化缓存稳定性。

## 五、新增文件及其作用说明

本章命令系统重构在 `com.bingtangcode.command` 包下新增了以下 8 个核心 Java 文件，它们各司其职，形成了高内聚、低耦合的架构设计：

| 新增文件名 | 所在路径 | 核心职责 |
|------|--------|--------|
| **CommandType.java** | `src/main/java/com/bingtangcode/command/` | **命令模式枚举**。定义了 `LOCAL`（只读本地）、`LOCAL_UI`（影响 UI 及会话状态）、`PROMPT`（注入模板并自动唤醒大模型）三类执行模式。 |
| **CommandContext.java** | `src/main/java/com/bingtangcode/command/` | **命令执行上下文**。将当前命令执行所需的依赖（包括参数 `arguments`、`UIController`、`DialogueManager`、`AgentLoop`、`ConfigManager`、`CommandRegistry`）封装成一个不可变上下文对象传递给 Handler。 |
| **UIController.java** | `src/main/java/com/bingtangcode/command/` | **界面控制抽象接口**。定义了控制台清屏、发送用户/系统消息、切换 Plan 模式、读取会话 Token 指标以及列表单选等交互行为，实现逻辑与渲染引擎的完全解耦。 |
| **Command.java** | `src/main/java/com/bingtangcode/command/` | **命令元数据实体类**。封装单个斜杠命令的静态配置属性（名、别名列表、描述、用法、隐藏状态）及运行期执行回调句柄。 |
| **CommandHandler.java** | `src/main/java/com/bingtangcode/command/` | **命令执行函数式接口**。只有一个 `handle(CommandContext)` 回调方法，支持使用 Lambda 表达式声明命令的业务逻辑。 |
| **CommandRegistry.java** | `src/main/java/com/bingtangcode/command/` | **命令注册与补全中心**。使用读写锁保障线程安全，在启动期提供重名/别名防撞校验；运行时提供命令查找及 Tab 前缀补全。 |
| **ActionListBox.java** | `src/main/java/com/bingtangcode/command/` | **终端交互式单选列表框**。进入 Raw Mode 捕获上下键和回车/ESC，在终端通过局部 ANSI 光标清除和反显，提供无闪烁的选择交互菜单。 |
| **BuiltInCommands.java** | `src/main/java/com/bingtangcode/command/` | **预置命令声明配置类**。统一声明注册 12 个系统内置高频命令的具体业务逻辑，通过单一静态方法 `createBuiltInCommands()` 向上游装配器提供列表。 |
| **CommandSystemTest.java** | `src/test/java/com/bingtangcode/command/CommandSystemTest.java` | **单元测试类**。自动化覆盖了注册防重冲突抛异常、前缀不敏感查找、Tab 多重匹配包含别名、隐藏命令拦截及模拟 UI 执行等功能。 |

## 六、本章设计与 Claude Code 的区别与对比

本章节实现的命令拦截与分发系统在设计思路上借鉴了市面上先进的 AI 编程助手（如 Anthropic 的 `Claude Code`），但在底层架构和实现语言上做出了符合自身定位的独特取舍：

| 维度 | bingtangCode (本章实现) | Claude Code (参考对比) |
|------|------------------------|-----------------------|
| **开发语言与生态** | 基于 **Java 21**，底层集成 **JLine3** 库提供控制台输入读取和状态更新。 | 基于 **Node.js / TypeScript**，依靠 npm 生态的 `ink` 或自定义终端包进行 UI 渲染。 |
| **启动冲突校验** | **强 Fail-Fast 校验**：在 `Main` 启动装配阶段通过 `CommandRegistry` 强行阻断任何重名/别名碰撞，抛出异常阻断进程。 | **动态注册与后覆盖模式**：通常在 JS 模块加载过程中进行动态注册，冲突时常采用后面的覆盖前面，或者在具体命令解析时再行区分。 |
| **解耦设计 (DIP)** | **高度抽象的 UIController 接口**：所有的命令不感知控制台或终端库的具体 API，只面向接口，极易对其进行 Stub / Mock 测试。 | 多数命令行为与 `process.stdout` 或特定终端 UI 库（如 Ink 组件）混写，测试通常依赖于真实控制台流重定向。 |
| **交互列表实现** | **低开销的 ActionListBox 原生实现**：直接通过终端 Raw Mode 读取原生 ANSI 并重绘，没有任何第三方依赖包，性能高，体积极小。 | 采用 Ink (React for CLI) 进行状态驱动的 UI 重绘，虽然表现力极其丰富，但依赖较重，需要整个 JS 虚拟 DOM 运行时的开销。 |
| **斜杠命令分流** | **硬拦截架构**：任何以 `/` 开头的输入（包含拼写错误的斜杠命令）在本地分流器被彻底拦截，在本地报错或提示 `/help`，绝对不会投递给大模型。 | 同样实行前缀拦截。但其命令可能与命令行配置（config）存在更复杂的双向绑定逻辑。 |
| **命令分发类别** | 明确划分为 `LOCAL`、`LOCAL_UI` , `PROMPT` 三大执行模式，利用枚举将各命令的职责划分得非常清晰。 | 命令功能在 handler 内部通过代码自行处理 UI 改变或网络请求，没有在元数据层面强限制执行模式。 |
