# flow-guide.md · bingtangCode 系统提示工程

面向小白。读完你能理解：bingtangCode 的系统提示怎么从 3 行变成 7 个结构化模块、什么是稳定/动态分流、system-reminder 怎么在运行中悄悄注入指令、以及这些设计各自解决什么问题。

## 一、为什么要重做系统提示

### 改之前

系统提示是 Main.java 里硬编码的三行字符串：

```java
"你是 bingtangCode，一个终端 AI 编程助手。底层由 " + providerName
    + " 驱动，当前模型为 " + modelName + "。\n"
    + "工具调用规则：\n"
    + "- 只读工具互不干扰，可以同一轮并发调用。\n"
    + "- 副作用工具会修改文件系统，系统会串行执行。\n"
    + "- 同一轮多个工具调用视为互不依赖。需要先创建再验证时分两轮。";
```

这个版本能干活，但有三个问题：

| 问题 | 具体表现 |
|------|---------|
| **无结构** | 所有指令混在一起，加新规则只能往后面追加，没有"这部分是讲工具的、那部分是讲安全的" |
| **无缓存优化** | provider 名、model 名和固定指令混在同一个字符串里，每次启动整个字符串都不同，API 缓存没法稳定命中 |
| **无运行时注入** | Plan Mode 只能靠过滤工具集来约束模型——调 write_file 时 API 直接不给你这个工具。但模型**不知道自己处于 plan 模式**，被拒绝后可能反复尝试不同策略 |

### 改之后

```
┌──────────────────────────────────────────────┐
│ SYSTEM 消息（稳定，可缓存）                     │
│ ┌──────────────────────────────────────────┐ │
│ │ 一、身份    你是 bingtangCode...         │ │
│ │ 二、系统约束  只做要求的任务...            │ │
│ │ 三、任务模式  探索性→建议，明确→执行...    │ │
│ │ 四、动作执行  评估可逆性...               │ │
│ │ 五、工具使用  优先专用工具...             │ │
│ │ 六、语气风格  简洁中文...                 │ │
│ │ 七、文本输出  不叙述推理过程...           │ │
│ │ ---                                      │ │
│ │ 环境信息（会话级快照，末尾拼接）            │ │
│ └──────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
┌──────────────────────────────────────────────┐
│ USER 消息（含 <system-reminder>，不缓存）     │
│ ┌──────────────────────────────────────────┐ │
│ │ <system-reminder>Plan模式：仅可用只读工具 │ │
│ │ </system-reminder>                       │ │
│ │                                          │ │
│ │ 帮我读一下 pom.xml                        │ │
│ └──────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

核心变化：**稳定内容走 SYSTEM（可缓存前缀）、动态内容通过 `<system-reminder>` 标签走 USER 消息（按轮次注入）**。

## 二、三个新组件 + 怎么连起来

### 2.1 组件一览

| 组件 | 文件 | 职责 | 一句话 |
|------|------|------|--------|
| **SystemPromptBuilder** | `core/SystemPromptBuilder.java` | 组装 7 个固定模块 + 环境信息 | "把散装指令拼成结构化系统提示" |
| **SystemReminderManager** | `core/SystemReminderManager.java` | 管理 Plan Mode 提醒的注入频率 | "决定这一轮该说完整版还是短标签" |
| **AgentLoop（更新）** | `agent/AgentLoop.java` | 每轮调用前注入提醒、模式切换时通知 | "在正确的时间点把提醒塞进对话" |

### 2.2 启动时接线

```
Main.main()
  │
  ├─ new SystemPromptBuilder()                     → promptBuilder
  ├─ buildEnvInfo(projectRoot)                     → envInfo
  │     ├─ System.getProperty("os.name")            → platform
  │     ├─ System.getenv("SHELL")                   → shell
  │     ├─ LocalDate.now()                          → date
  │     └─ git status (执行命令抓快照)               → gitStatus
  │
  ├─ promptBuilder.build(providerName, modelName, envInfo)
  │     └─ 拼接: 身份 + 系统约束 + 任务模式 + ... + 环境信息
  │        → 传给 new DialogueManager(systemPrompt, ...)
  │
  ├─ new SystemReminderManager()                    → reminderManager
  │
  └─ new AgentLoop(dialogue, provider, registry, bus, 20, reminderManager)
        └─ AgentLoop 持有 reminderManager 引用，后续每轮调它
```

### 2.3 为什么 EnvInfo 放在 SYSTEM 末尾而不担心缓存

环境信息（工作目录、git 状态）是会话级快照——每次启动时获取一次。它放在 SYSTEM 消息的**最后几行**。

Anthropic API 的缓存是**前缀匹配**：只要前面的 7 个固定模块没变，缓存就能命中到环境信息之前。环境信息之后的内容（包括它自己）不走缓存，但因为只有几行，成本忽略不计。

```
缓存命中:
  SYSTEM: [7个模块 ..........................] [环境信息]
          ←── 缓存命中 ──→                  ←── 不缓存，但只有几行
```

如果后续发现 git 状态每轮变化导致缓存失效，可以把动态字段从 SYSTEM 移到 `<system-reminder>` 消息中——现在留了这条路，只是先不做。

## 三、SystemPromptBuilder — 怎么拼出系统提示

### 3.1 接口

```java
public class SystemPromptBuilder {

    public record EnvInfo(
        String workDir, String platform, String shell,
        String osVersion, String date, String gitStatus
    ) {}

    public String build(String providerName, String modelName, EnvInfo env);
    // 返回完整系统提示字符串
}
```

### 3.2 内部结构

```java
public String build(...) {
    sb.append(buildIdentity(providerName, modelName));  // 一、身份
    sb.append("\n\n");
    sb.append(buildConstraints());                       // 二、系统约束
    sb.append("\n\n");
    sb.append(buildTaskMode());                          // 三、任务模式
    sb.append("\n\n");
    sb.append(buildActionExecution());                   // 四、动作执行
    sb.append("\n\n");
    sb.append(buildToolUsage());                         // 五、工具使用
    sb.append("\n\n");
    sb.append(buildTone());                              // 六、语气风格
    sb.append("\n\n");
    sb.append(buildTextOutput());                        // 七、文本输出
    sb.append("\n\n");
    sb.append(buildEnvSection(env));                     // 环境信息
    return sb.toString();
}
```

每个模块一个 private 方法，内部分别 return 一个文本块。加新模块只需三件事：写一个 `build*()` 方法、在 `build()` 里加一行 `sb.append`、在两个模块之间加 `"\n\n"`。

### 3.3 七个模块各自写什么

| 模块 | 核心规则数 | 解决什么问题 |
|------|:--------:|-------------|
| 一、身份 | 1 句 | 让模型知道自己是谁 |
| 二、系统约束 | 5 条 | 不越权、不瞎猜、不画蛇添足 |
| 三、任务模式 | 5 条 | 什么情况自己干、什么情况问用户、干完怎么汇报 |
| 四、动作执行 | 4 条 | git 安全、可逆性评估、不走破坏性捷径 |
| 五、工具使用 | 5 条 | 专用工具优先、编辑前必读、并发/串行规则 |
| 六、语气风格 | 4 条 | 中文、简洁、不 emoji、引用用 file:line 格式 |
| 七、文本输出 | 4 条 | 不叙述推理、关键节点给更新、一句话够 |

### 3.4 设计原则

**稳定 vs 变化**

七个固定模块的内容很少变动——身份永远是 bingtangCode，工具规则不会因为会话不同而改变。这些放在 SYSTEM 消息里，API 可以缓存。

**没有放进 SYSTEM 的东西**

- 模式切换指令（Plan/Full）→ 走 `<system-reminder>` 消息通道 ✅
- 自定义指令（将来 `/set` 命令）→ 走 `<system-reminder>` 消息通道 🔲
- 已激活的 Skill 列表 → 走 `<system-reminder>` 消息通道 🔲
- 长期记忆 → 走 `<system-reminder>` 消息通道 🔲

> ✅ = 已实现　　🔲 = 未实现（Out of Scope，后续章节做）

这些东西随时可能变，放 SYSTEM 里会频繁破坏缓存前缀。

## 四、SystemReminderManager — 怎么控制注入频率

### 4.1 它解决的问题

Plan Mode 下模型只能用只读工具。改之前，模型**不知道为什么**有些工具调不了——它看到 tools[] 里没有 write_file，但不知道"不是我漏了，是你切到了 Plan Mode"。被拒绝后可能反复尝试不同写法。

现在每轮对话前注入一条 `<system-reminder>` 消息：模型看到 `<system-reminder>` 标签就知道这是系统指令，不是用户说的话，不会去回复它。

### 4.2 为什么是两层：硬约束 + 软引导

Plan 模式的设计在**两个不同层面**同时生效，各自解决不同的问题：

```
Layer A: 工具过滤（硬约束，AgentLoop.selectTools）
  └─ API 请求的 tools[] 里只放 read_file、find_files、search_content
  └─ 模型想调 write_file 也调不了——API 直接不认这个工具名
  └─ 解决的问题：物理上杜绝写操作，不可绕过

Layer B: 系统提醒（软引导，SystemReminderManager）
  └─ 第一轮：Plan模式：仅可用只读工具，专注于分析问题、制定计划，不要尝试修改文件或执行命令
  └─ 后续每 3 轮：同上完整提醒
  └─ 其余轮次：Plan模式：仅可用只读工具（精简版）
  └─ 解决的问题：告诉模型「为什么你只有这 3 个工具」
```

**两层缺一不可：**

| | 只有硬约束（无提醒） | 只有软提醒（无过滤） |
|---|---|---|
| 模型看到 | 工具列表里只有 3 个只读工具 | 6 个工具全在 |
| 模型会怎么做 | 不知道原因，可能以为系统加载失败，反复碰壁或编造不存在的工具 | 偶尔会忽略提醒，直接调 write_file |
| 结果 | 浪费 token，用户困惑 | 偶尔写出文件，Plan 模式形同虚设 |

两层叠起来：工具过滤是**物理锁**——想调也调不了。系统提醒是**说明书**——告诉你锁在哪、为什么锁、怎么配合。这种设计在 Claude Code 里也是同一套逻辑。

### 4.3 接口和状态（已更新）

```java
public class SystemReminderManager {

    private AgentLoop.Mode currentMode;  // 当前 PLAN 还是 FULL
    private int roundCount;              // Plan 模式下经过了几轮
    private boolean firstRound;          // 是否刚切到 Plan 模式

    void onModeSwitch(Mode mode);  // 模式切换时调用，重置计数器
    String getReminder();          // 返回本轮该注入的提醒（FULL 模式返回 null）
    void onRoundComplete();        // 每轮结束后调用，计数器 +1
}
```

### 4.4 频率规则（N=3）

```
切到 Plan Mode:
  onModeSwitch(PLAN) → roundCount=0, firstRound=true

第 1 轮: firstRound=true  → FULL   → roundCount=1
第 2 轮: roundCount=1     → SHORT  → roundCount=2
第 3 轮: roundCount=2     → SHORT  → roundCount=3
第 4 轮: 3%3==0           → FULL   → roundCount=4
第 5 轮: roundCount=4     → SHORT  → roundCount=5
第 6 轮: roundCount=5     → SHORT  → roundCount=6
第 7 轮: 6%3==0           → FULL   → ...
```

FULL 和 SHORT 的具体内容：

```
FULL:  <system-reminder>Plan模式：仅可用只读工具，专注于分析问题、制定计划，
       不要尝试修改文件或执行命令。</system-reminder>

SHORT: <system-reminder>Plan模式：仅可用只读工具</system-reminder>
```

切回 FULL 模式时 `getReminder()` 返回 `null`——Do 模式是全工具可用，不需要额外提醒。

### 4.5 为什么是 3 轮

- **1 轮**：每条消息都塞完整提醒，浪费 token，而且占用模型注意力
- **不重复**：只首轮提醒一次，模型在多轮对话中可能"忘记"这个约束（context 长了之后注意力稀释）
- **3 轮**：折中。首轮完整提醒让模型明确知道自己受限，中间两轮短标签维持记忆但不占太多 token，第 4 轮再次完整提醒刷新约束

## 五、注入流程 — 一次完整调用

以 Plan Mode 为例，用户输入两条连续指令。

### 阶段 A：用户切到 Plan Mode

```
SessionManager.start()
  └─ 用户输入 "/plan"
       ├─ agentLoop.setMode(AgentLoop.Mode.PLAN)
       │    ├─ this.mode = PLAN                    ← 字段赋值
       │    └─ reminderManager.onModeSwitch(PLAN)  ← roundCount=0, firstRound=true
       │
       └─ System.out.println("已切换到 Plan Mode，仅可用只读工具")
          continue  ← 不调 run()
```

### 阶段 B：用户输入第一条指令

```
SessionManager.start()
  └─ 用户输入 "帮我读一下 pom.xml"
       └─ agentLoop.run("帮我读一下 pom.xml")
```

### 阶段 C：AgentLoop — 首轮注入

```
AgentLoop.run("帮我读一下 pom.xml")
  │
  ├─ dialogue.addUserMessage("帮我读一下 pom.xml")
  │     → history = [SYSTEM(7模块+环境), USER("帮我读...")]
  │
  ├─ iteration = 1
  │
  ├─ injectReminder()
  │     ├─ reminderManager.getReminder()
  │     │     ├─ currentMode == PLAN  ✓
  │     │     ├─ firstRound == true   ✓
  │     │     ├─ firstRound = false   ← 消费标记
  │     │     └─ return FULL_REMINDER
  │     │
  │     └─ dialogue.addMessage(USER(
  │          "<system-reminder>Plan模式：仅可用只读工具，专注于分析问题、制定计划，"
  │          + "不要尝试修改文件或执行命令。</system-reminder>"
  │        ))
  │           → history = [
  │                SYSTEM(7模块+环境),
  │                USER("<system-reminder>Plan模式...</system-reminder>"),
  │                USER("帮我读...")
  │              ]
  │
  └─ runRoundWithRetry() → dialogue.doRound(...)
       │
       ├─ buildApiMessages()
       │     合并连续 USER:
       │       USER("<system-reminder>...</system-reminder>\n帮我读...")
       │     → 发送给 LLM 的 messages:
       │       [SYSTEM(7模块+环境), USER("<system-reminder>Plan模式...</system-reminder>\n帮我读...")]
       │
       └─ LLM 看到:
            - system prompt 里有身份、约束、工具规则
            - <system-reminder> 告诉它现在是 Plan 模式
            - 用户让它读文件
            → 调 read_file("pom.xml") → 正常执行 → 返回结果
```

### 阶段 D：AgentLoop — 第二轮（短标签）

```
AgentLoop.run() 继续:
  │
  ├─ result.completed() == false → 继续
  ├─ reminderManager.onRoundComplete() → roundCount = 1
  │
  ├─ iteration = 2
  ├─ injectReminder()
  │     ├─ reminderManager.getReminder()
  │     │     ├─ firstRound == false
  │     │     ├─ roundCount(1) % 3 != 0
  │     │     └─ return SHORT_REMINDER
  │     │
  │     └─ dialogue.addMessage(USER(
  │          "<system-reminder>Plan模式：仅可用只读工具</system-reminder>"
  │        ))
  │
  └─ runRoundWithRetry() → dialogue.doRound(...)
       │
       ├─ buildApiMessages()
       │     history 最后几条:
       │       ... ASSISTANT("好的", [read_file])
       │           USER("", [], [ToolResult("pom.xml内容是...")])
       │           USER("<system-reminder>Plan模式：仅可用只读工具</system-reminder>")
       │     → 合并后:
       │       USER("<system-reminder>Plan模式：仅可用只读工具</system-reminder>\n\n"
       │            + [流水号] pom.xml内容是...")
       │
       └─ LLM 看到短标签 + 工具结果
            → 知道自己还在 Plan 模式
            → 总结文件内容，不调写工具
```

### 阶段 E：用户切回 Do Mode

```
SessionManager.start()
  └─ 用户输入 "/do"
       ├─ agentLoop.setMode(AgentLoop.Mode.FULL)
       │    ├─ this.mode = FULL
       │    └─ reminderManager.onModeSwitch(FULL)
       │         ├─ currentMode = FULL
       │         ├─ firstRound = false
       │         └─ roundCount = 0
       │
       └─ System.out.println("已切换到 Do Mode，全工具可用")

下一次 agentLoop.run():
  injectReminder()
    └─ reminderManager.getReminder()
         └─ currentMode == FULL → return null  ← 不注入
```

整个 Do Mode 期间 `getReminder()` 都返回 null，历史中不会出现 `<system-reminder>` 消息。

## 六、关键设计决策

### 6.1 稳定内容走 SYSTEM，动态内容走 USER 消息

```
SYSTEM 消息（可缓存前缀）:
  ✓ 身份、约束、任务模式、执行规则、工具规则、风格、输出规则
  ✓ 环境信息（会话快照，只在末尾几行）

USER 消息（不缓存，按轮次注入）:
  ✓ Plan/Full 模式提醒（<system-reminder>）
  ✓ 自定义指令（将来）
  ✓ Skill 列表（将来）
  ✓ 长期记忆（将来）
```

缓存原理：Anthropic API 对请求体做前缀匹配。只要 SYSTEM 消息的前 N 个字符不变，那些 token 就不用重新计算。7 个固定模块每次启动完全一致，API 可以直接从缓存读它们的 KV 向量，省掉处理时间。

动态内容（模式提醒、自定义指令）放在 USER 消息里——它们出现在 SYSTEM 之后，不会破坏缓存前缀。

### 6.2 为什么用 `<system-reminder>` 标签而不是 SYSTEM 角色

SYSTEM 角色的消息位置在最前面（Anthropic API 要求 system 消息在所有 user/assistant 之前），无法在对话中途插入。

`<system-reminder>` 标签放在 USER 角色消息里，可以：
- **在任意轮次插入**——模式切换后立即生效，不需要重新发 system 消息
- **被模型识别为指令**——模型训练数据中这个标签的语义是"系统指令，不要当作用户对话"
- **不污染缓存**——USER 消息本来就不在缓存前缀内

### 6.2.1 为什么不需要改 Provider

`<system-reminder>` 只是 USER 消息正文里的一段文本字符串，对 Provider 来说和"帮我读一下 pom.xml"没有任何区别——它只是一串字符。Provider 的工作是把 `Message(Role.USER, content)` 原样序列化成 API 请求体里的 JSON 字段，不关心 content 里写的是中文、英文还是 XML 标签。

模型认识这个标签靠的是训练数据中的语义约定，不是协议级别的特殊通道。从 bingtangCode 的角度，整条链路没有一行 Provider 代码被改动：

```
SystemReminderManager 生成文本
  → AgentLoop 包成 Message(Role.USER, "<system-reminder>...</system-reminder>")
  → DialogueManager.buildApiMessages() 合并到相邻 USER 消息
  → Provider.streamChat() 原样塞进 JSON 的 content 字段
  → API 收到标准请求，模型自己理解标签语义
```

对比：如果走 SYSTEM 角色注入，就需要 Provider 支持多段 system 消息或中途插入 system 消息——Anthropic API 不允许 system 出现在 user/assistant 之后，OpenAI 虽然支持但语义不同。用 `<system-reminder>` 绕过了所有协议差异。

### 6.3 为什么 buildApiMessages 合并连续 USER 消息是对的

```java
// DialogueManager.buildApiMessages() 第 144-153 行
if (size > 0 && cleaned.get(size - 1).role() == msg.role()
               && msg.role() != Role.SYSTEM) {
    // 合并连续同角色消息
    cleaned.set(size - 1, new Message(prev.role(),
            prev.content() + "\n" + msg.content(), ...));
}
```

`<system-reminder>` 消息是 USER 角色，用户的输入也是 USER 角色，连续出现自动合并。LLM 看到：

```
USER: <system-reminder>Plan模式：仅可用只读工具</system-reminder>
      帮我读一下 pom.xml
```

一个 USER 消息里同时有系统指令和用户输入。模型把 `<system-reminder>` 部分当约束，把后面部分当任务，不会去回复标签本身。

### 6.4 提醒在 doRound 之前注入、而不是 doRound 返回之后

```
✓:  injectReminder() → doRound() → onRoundComplete()
✗:  doRound() → onRoundComplete() → injectReminder()
```

原因是 `doRound()` 调用 `buildApiMessages()` 合并消息——如果提醒在 doRound 之后注入，合并时就会把**下一轮**的工具结果和**这一轮**的提醒并在一起。先注入提醒再调 doRound，保证本轮 LLM 就看到当前的模式约束。

### 6.5 双重强化：系统提示 + 工具描述

关键规则同时在两处声明：

| 规则 | 系统提示（五、工具使用） | 工具描述 |
|------|------------------------|---------|
| 编辑前必先读 | "编辑文件前必须先 Read 过目标文件" | EditFileTool: "注意：调用本工具前必须先使用 read_file 读取目标文件" |
| 修改优先 edit 而非 write | "优先用专用工具：Read 读文件、Edit 改文件、Write 写文件" | WriteFileTool: "注意：对于已有文件的修改，优先使用 edit_file 而非 write_file" |

系统提示给的是通用原则，工具描述给的是调用该工具时的具体约束。两者互补——模型在做决策时从系统提示学规则，在准备调工具时从工具描述看到硬性提醒。

## 八、环境信息是怎么获取的

```java
// Main.java
private static EnvInfo buildEnvInfo(Path projectRoot) {
    String workDir = projectRoot.toAbsolutePath().toString();
    String platform = System.getProperty("os.name", "未知");
    String shell = System.getenv("SHELL");
    if (shell == null || shell.isEmpty()) shell = "未知";
    String osVersion = System.getProperty("os.version", "未知");
    String date = LocalDate.now().toString();
    String gitStatus = getGitStatus(projectRoot);
    return new EnvInfo(workDir, platform, shell, osVersion, date, gitStatus);
}

private static String getGitStatus(Path projectRoot) {
    try {
        ProcessBuilder pb = new ProcessBuilder("git", "status");
        pb.directory(projectRoot.toFile());
        pb.redirectErrorStream(true);
        Process process = pb.start();
        // 读 stdout，等进程结束
        try (BufferedReader reader = ...) {
            String result = reader.lines().collect(Collectors.joining("\n"));
            process.waitFor();
            if (process.exitValue() == 0) return result;
        }
    } catch (Exception ignored) {}
    return "未知";    // git 不可用时优雅降级
}
```

`getGitStatus` 失败时不会让整个程序崩溃——返回"未知"继续启动。这在非 git 目录或 CI 环境里是必要的。

## 十、缓存验证方案（设计草案，待实施）

### 10.1 要验证什么

把 7 个固定模块放进 SYSTEM 消息是为了让 API 缓存命中——同样的提示词前缀不需要重复计算。但策略是否真的生效，目前是黑盒。缓存验证要回答三个问题：

| 问题 | 验证方式 |
|------|---------|
| 缓存是否命中？ | 看 API 返回的 `cache_read_input_tokens` 是否 > 0 |
| 命中率多少？ | 同一会话内，第 2 轮起缓存命中 token 数占 input token 的百分比 |
| 环境信息是否破坏缓存？ | 对比两个不同目录的会话，看缓存是否仍然命中（前缀匹配应跨会话生效） |

### 10.2 API 返回哪些缓存字段

**Anthropic**（SSE 流）

在 `message_start` 事件的 `message.usage` 中：

```json
{
  "type": "message_start",
  "message": {
    "usage": {
      "input_tokens": 1200,
      "cache_read_input_tokens": 800,       // ← 从缓存读取的 token 数
      "cache_creation_input_tokens": 400     // ← 新写入缓存的 token 数
    }
  }
}
```

- `cache_read_input_tokens`：本次请求有多少 token 直接读了缓存（省了计算）
- `cache_creation_input_tokens`：本次请求有多少 token 被写入了缓存（下次能命中）

**OpenAI**（SSE 流）

在流式响应的 `usage` 块中：

```json
{
  "usage": {
    "prompt_tokens": 1200,
    "completion_tokens": 150,
    "prompt_tokens_details": {
      "cached_tokens": 800                   // ← 从缓存命中的 token 数
    }
  }
}
```

- OpenAI 只区分"缓存命中了多少"，不区分"读缓存 vs 写缓存"
- 对应关系：`cacheReadTokens = cached_tokens`，`cacheCreationTokens = 0`

### 10.3 需要改动的文件和内容

#### ① StreamCallback — 新增回调

```java
// StreamCallback.java — 新增方法
default void onCacheUsage(int cacheReadTokens, int cacheCreationTokens) {}
```

默认空实现，不影响已有调用方。

#### ② AgentEvent.TokenUsage — 扩展字段

```java
// AgentEvent.java — 当前（5 字段）
record TokenUsage(int inputTokens, int outputTokens,
                  int totalInput, int totalOutput)

// 扩展为（7 字段）
record TokenUsage(int inputTokens, int outputTokens,
                  int totalInput, int totalOutput,
                  int cacheReadTokens, int cacheCreationTokens)
      implements AgentEvent {}
```

`DialogueManager.doRound()` 中创建 TokenUsage 的位置补上两个新字段（默认从 StreamCallback 收值后填入）。

#### ③ AnthropicProvider — 提取缓存字段

修改 `message_start` 事件解析（当前 `AnthropicProvider.java:251-253`）：

```java
// 现有代码只提取 input_tokens
Map<String, Object> usage = (Map<String, Object>) message.get("usage");
if (usage != null && usage.get("input_tokens") instanceof Number n) {
    inputTokens = n.intValue();
}

// 新增：提取缓存字段
int cacheReadTokens = 0;
int cacheCreationTokens = 0;
if (usage != null) {
    if (usage.get("cache_read_input_tokens") instanceof Number n) {
        cacheReadTokens = n.intValue();
    }
    if (usage.get("cache_creation_input_tokens") instanceof Number n) {
        cacheCreationTokens = n.intValue();
    }
}
```

在 `message_stop` 处（`AnthropicProvider.java:323`），`callback.onUsage()` 之后调用：

```java
callback.onUsage(inputTokens, outputTokens);
callback.onCacheUsage(cacheReadTokens, cacheCreationTokens);  // 新增
callback.onComplete();
```

#### ④ OpenAIProvider — 提取缓存字段

修改 `usage` 对象解析（当前 `OpenAIProvider.java:215-218`）：

```java
Map<String, Object> usage = (Map<String, Object>) event.get("usage");
if (usage != null) {
    if (usage.get("prompt_tokens") instanceof Number n) inputTokens = n.intValue();
    if (usage.get("completion_tokens") instanceof Number n) outputTokens = n.intValue();

    // 新增：提取缓存字段
    Map<String, Object> details = (Map<String, Object>) usage.get("prompt_tokens_details");
    if (details != null && details.get("cached_tokens") instanceof Number n) {
        cacheReadTokens = n.intValue();
    }
}
```

在两个 `onUsage` 回调点之后（`OpenAIProvider.java:208` 和 `OpenAIProvider.java:255`）都调用 `callback.onCacheUsage(cacheReadTokens, cacheCreationTokens)`。`cacheCreationTokens` 始终为 0。

#### ⑤ DialogueManager.doRound() — 传递缓存值

StreamCallback 的 `onUsage` 和 `onCacheUsage` 在不同时机回调（都在同一轮内）。需要在 doRound 中用局部变量暂存缓存值，在 `onTokenUsage` 事件创建时填入：

```java
// doRound() 内部
AtomicInteger cacheRead = new AtomicInteger(0);
AtomicInteger cacheCreation = new AtomicInteger(0);

StreamCallback collector = new StreamCallback() {
    @Override
    public void onCacheUsage(int cr, int cc) {
        cacheRead.set(cr);
        cacheCreation.set(cc);
    }
    // ... 其他回调不变
};
```

创建 TokenUsage 事件时：

```java
bus.fire(new AgentEvent.TokenUsage(
    inputTokens, outputTokens, totalInput.get(), totalOutput.get(),
    cacheRead.get(), cacheCreation.get()));
```

#### ⑥ TuiEventListener — 终端展示

当前 TokenUsage 展示格式：

```
  ↑1.2k ↓150 · 累计 1.4k
```

改为：

```
  ↑1.2k ↓150 · 缓存 800 · 累计 1.4k
```

或命中为 0 时不显示缓存部分，避免无用信息：

```
  ↑1.2k ↓150 · 累计 1.4k          ← cacheReadTokens == 0 时不展示缓存
  ↑1.2k ↓150 · 缓存 800 · 累计 1.4k  ← 命中时展示
```

### 10.4 验证场景

| # | 场景 | 操作 | 预期 |
|---|------|------|------|
| 1 | 同一会话第 1 轮 | 启动 bingtangCode，输入"你好" | `cacheReadTokens` = 0（首次请求，无缓存） |
| 2 | 同一会话第 2 轮 | 接着输入"帮我读 pom.xml" | `cacheReadTokens` > 0（7 个模块命中缓存），本应缓存约 800+ token |
| 3 | 同一会话更多轮 | 继续第 3、4 轮 | 缓存命中稳定在相似数值 |
| 4 | `/plan` 切换后 | 输入 `/plan`，再发任务 | 缓存仍然命中（SYSTEM 消息没变，reminder 是 USER 消息不影响） |
| 5 | 重启会话 | 退出再启动，输入"你好" | `cacheReadTokens` > 0（跨会话缓存，因为 SYSTEM 消息未变） |
| 6 | 不同目录 | 在另一个目录启动 bingtangCode | `cacheReadTokens` 可能 < 场景 5（环境信息不同导致 SYSTEM 消息尾部变化，但 7 个模块仍应命中） |

### 10.5 定性评估对照

不做自动化评估，用以下对照人工判断：

| 对照组 | 条件 | 观察指标 |
|--------|------|---------|
| 基线 | 当前 bingtangCode（无缓存分流，3 行 system prompt） | 查看 API 返回的原始响应，记录 `cache_read_input_tokens` |
| 优化后 | 实施本章改动后 | 同一场景下 `cache_read_input_tokens` 有明显非零值 |
| 环境变化 | 修改 bingtangCode 所在目录后重新启动 | 观察缓存命中是否下降（环境信息变了） |

### 10.6 不做什么

- 不做自动化缓存命中率监控（Out of Scope）
- 不做缓存命中率可视化图表
- 不做跨模型的缓存策略对比
- 不做缓存预热（手动触发写缓存）
