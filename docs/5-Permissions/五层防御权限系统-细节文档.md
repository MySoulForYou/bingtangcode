面向小白。读完你能理解：bingtangCode 如何保护你的项目安全、五层防御各自做什么、规则怎么写、人在回路怎么交互。

## 一、权限系统解决什么问题

### 之前的问题

- 路径越界检查硬编码在 6 个工具里，代码重复
- Bash 命令执行前有个简单的 y/N 确认弹窗，但**10 秒超时自动拒绝**
- 没有可配置的规则——每次都要手动确认，无法沉淀信任
- 没有分级信任——要么全信，要么全不信
- 同一个操作确认过了，下次还得再确认

### 现在的能力

| 能做 | 不能做 |
|------|--------|
| 5 层防御自动拦截 | 网络请求限制 |
| YAML 文件配置规则 | 资源配额（CPU/内存） |
| 4 档权限模式一键切换 | 审计日志 |
| 人在回路箭头选确认/拒绝 | MCP 工具权限 |
| 永久放行写入本地文件 | 规则热加载 |
| 规则拒绝后 Agent 继续运行 | 多用户/角色权限 |

## 二、五层防御架构

```
用户输入 → AgentLoop.run()
               │
               ▼
         selectTools()  ← 根据 PermissionMode 过滤工具列表
               │
               ▼
         dialogue.doRound(tools, ...)
               │
               ▼
         executeToolBatch(toolCalls)
               │
               ▼
    ┌──────────────────────────┐
    │  PermissionGate.check()  │  ← 五层防御集中拦截点
    │                          │
    │  Layer 1: 黑名单 (硬编码)  │     匹配 → DENY（不可绕过）
    │  Layer 2: 路径沙箱        │     越界 → DENY
    │  Layer 3: 规则引擎        │     匹配 → ALLOW / DENY
    │  Layer 4: 权限模式兜底    │     未命中 → 模式决定 Allow/Ask
    │  Layer 5: 人在回路        │     Ask → 用户交互确认
    └──────────────────────────┘
               │
          ALLOW ▼
         ToolExecutor.execute()
               │
          DENY ▼
         返回结构化错误 ToolResult → 模型调整策略继续循环
```

### 新增模块一览

| 模块 | 文件 | 一句话 |
|------|------|--------|
| **PermissionMode** | `permission/PermissionMode.java` | 4 档模式枚举 + 默认行为决策 |
| **PermissionRule** | `permission/PermissionRule.java` | 一条规则：工具名 + 模式 + 允许/拒绝 + 来源 |
| **PermissionResult** | `permission/PermissionResult.java` | 检查结果：放行/拒绝 + 原因 + 匹配规则 |
| **Blacklist** | `permission/Blacklist.java` | 19 条高危命令正则，硬编码不可绕过 |
| **PathSandbox** | `permission/PathSandbox.java` | 路径校验 + 符号链接解析 |
| **RuleEngine** | `permission/RuleEngine.java` | 3 层规则（本地→项目→用户）+ glob 匹配 |
| **PermissionGate** | `permission/PermissionGate.java` | 五层防御编排者 |
| **PermissionPrompt** | `permission/PermissionPrompt.java` | 人在回路终端 UI |
| **PermissionConfigLoader** | `permission/PermissionConfigLoader.java` | 三层 YAML 配置加载 |
| **PermissionConfigManager** | `permission/PermissionConfigManager.java` | 默认模式加载 + ALLOW_FOREVER 持久化 |
| **ToolFriendlyName** | `permission/ToolFriendlyName.java` | 内部名→友好名映射 + 主参数提取 |

### 修改的文件

| 文件 | 改动 |
|------|------|
| `AgentLoop.java` | 删除 `Mode` 枚举，改用 `PermissionMode`，实现 `PermissionModeProvider` |

## 三、一次完整的权限检查流程

以"模型要执行 `git push --force origin main`"为例。

### 阶段 A：工具调用到达 DialogueManager

```
DialogueManager.executeToolBatch(toolCalls)
  └─ executeOne(execute_command("toolu_001", {command:"git push --force origin main"}))
       │
       ├─ eventBus.fire(ToolCallStarted("execute_command", "toolu_001"))
       │
       └─ permissionGate.check(tc)
```

### 阶段 B：Layer 1 — 黑名单

```
PermissionGate.check(tc)
  ├─ isBash = true（execute_command）
  ├─ 提取主参数 command = "git push --force origin main"
  │
  └─ blacklist.check("git push --force origin main")
       ├─ 遍历 19 条正则
       ├─ Pattern "git\s+push\s+--force\s+.*main" 命中！
       └─ return PermissionResult.deny(
             "匹配黑名单: git\s+push\s+--force\s+.*main",
             "blacklist")
```

**结果是 DENY。** 后续 Layer 2~5 全部跳过，直接返回结构化错误。

### 阶段 C：如果是合法操作——`git status`

```
Layer 1: blacklist.check("git status")
  └─ 19 条正则全不命中 → allow()，继续

Layer 2: 跳过（Bash 不经过路径沙箱）

Layer 3: ruleEngine.check("execute_command", {command:"git status"}, projectRoot)
  ├─ ToolFriendlyName → friendlyName = "Bash", mainParam = "git status"
  ├─ 匹配 localRules → 假设有 "Bash(git *)" → allow
  │   └─ matchesPattern("git *", "git status")
  │        ├─ pattern 含 "*" → glob 匹配
  │        └─ PathMatcher("glob:git *").matches(Path.of("git status")) → true
  └─ return PermissionResult.allow()

结果是 ALLOW，工具正常执行。
```

### 阶段 D：如果没有匹配规则——进入人在回路

```
Layer 4: 当前模式 = DEFAULT
  ├─ isBash = true, isReadOnly = false
  ├─ getDefaultAction(false, true) → ASK
  └─ 进入 Layer 5

Layer 5: hitlHandler.ask(tc, DEFAULT)
  └─ PermissionPrompt.ask()
       ├─ 打印选项块：
       │   ┌─ 权限确认 ───────────────────────────┐
       │   │                                      │
       │   │  Bash: git status                    │
       │   │  未匹配任何规则，是否允许？            │
       │   │                                      │
       │   │  ▸ 1. 允许本次         (ALLOW_ONCE)   │
       │   │    2. 永久允许         (ALLOW_FOREVER) │
       │   │    3. 拒绝本次         (DENY_ONCE)     │
       │   │                                      │
       │   │  快捷键: [1/y] 允许  [3/n/d] 拒绝     │
       │   │  ↑/k 上移  ↓/j 下移  Enter 确认      │
       │   └──────────────────────────────────────┘
       │
       └─ 阻塞等待按键...
           用户按 "2" → AskResult.ALLOW_FOREVER
           
PermissionGate 收到 ALLOW_FOREVER:
  ├─ 构造 PermissionRule("Bash", "git status", ALLOW, "local")
  ├─ configManager.appendRuleToLocal(rule)
  │    └─ 写入 .bingtangcode/permissions.local.yaml:
  │         - tool: Bash
  │           pattern: "git status"
  │           action: allow
  └─ return PermissionResult.allow()

结果是 ALLOW，工具执行，规则持久化。
```

## 四、五层防御详解

### 4.1 Layer 1：黑名单

19 条硬编码正则，编译后静态缓存：

```
rm -rf /       → 强删根目录
rm -rf ~       → 强删家目录
rm -rf *       → 强删当前目录
sudo           → 提权前缀
curl | bash    → curl 管道 shell
wget -O- | bash→ wget 管道 shell
chmod 777      → 全开放权限
chmod -R 777   → 递归全开放
chown -R       → 递归改所有者
mkfs.          → 格式化文件系统
dd if=         → 磁盘直接读写
>/dev/sd[a-z]  → 覆盖磁盘设备
fork 炸弹       → :(){ :|:& };:
shutdown       → 关机
reboot         → 重启
init [0-6]     → 切换运行级别
systemctl stop/disable/mask → 停系统服务
git push --force main   → force push main
git push --force master → force push master
```

**不可绕过**：无论在什么模式（包括 BYPASS_PERMISSIONS），黑名单始终生效。

### 4.2 Layer 2：路径沙箱

```java
PathSandbox.validate(projectRoot, filePath)
  1. projectRoot.resolve(filePath)  → 初步路径
  2. toRealPath()  → 解析所有符号链接拿真实路径
  3. realPath.startsWith(projectRoot.toRealPath())  → 前缀判断
  4. 越界 → 抛 PathViolationException
```

**关键增强**：符号链接解析。如果在项目内创建指向 `/etc/passwd` 的软链接，旧代码的 `resolve+normalize+startsWith` 会放行（因为链接路径在项目内），新代码的 `toRealPath()` 会解析到真实的 `/etc/passwd` 然后拒绝。

**路径不存在的处理**：写新文件时路径还不存在，`toRealPath()` 会抛异常。PathSandbox 会逐级向上找存在的祖先目录来验证。

### 4.3 Layer 3：规则引擎

#### 优先级体系

```
对每个工具调用:
  ① 查 permissions.local.yaml    → 命中则返回
  ② 查 permissions.yaml（项目）   → 命中则返回
  ③ 查 permissions.yaml（用户）   → 命中则返回
  ④ 所有层无命中 → null（未决定）
```

#### 同层规则内的优先级

```
1. deny 优先——两条都命中时 deny 获胜
2. 同 effect 按书写顺序——第一个命中的 allow 先生效
```

#### 匹配语义

- pattern 含 `*` 或 `**` → glob 匹配（`*` 单层，`**` 递归）
- pattern 不含 `*` → 精确字符串匹配
- 文件类工具：主参数自动转为项目相对路径后再匹配
- Bash：直接对命令字符串匹配

### 4.4 Layer 4：模式兜底

| 模式 | 只读工具 | 写工具 | Bash | 说明 |
|------|---------|--------|------|------|
| DEFAULT | Allow | Ask | Ask | 默认最安全 |
| ACCEPT_EDITS | Allow | Allow | Ask | 信任代码编辑 |
| PLAN | Allow | Ask（工具不可见） | Ask（工具不可见） | 只暴露只读工具 |
| BYPASS_PERMISSIONS | Allow | Allow | Allow | 不走 Ask，黑名单沙箱仍拦截 |

### 4.5 Layer 5：人在回路

```
PermissionPrompt.ask(tc, mode)
  ├─ 暂停 token 流（阻塞 Agent Loop 当前迭代）
  ├─ 打印多行选项块（含工具名、主参数）
  ├─ 循环读取按键：
  │    ↑/k → 上移光标
  │    ↓/j → 下移光标
  │    Enter → 确认当前高亮
  │    1/y → ALLOW_ONCE
  │    2   → ALLOW_FOREVER
  │    3/n/d → DENY_ONCE
  └─ 清除选项块 → 返回 AskResult
```

**无超时**：等待用户明确选择，不会超时自动拒绝。

**ALLOW_FOREVER 的持久化路径**：
```
PermissionGate.check()
  → hitlHandler.ask() 返回 ALLOW_FOREVER
  → 构造 PermissionRule("Bash", "git status", ALLOW, "local")
  → configManager.appendRuleToLocal(rule)
       └─ 写入 .bingtangcode/permissions.local.yaml
            文件不存在则创建含 rules: 的空骨架
            追加 YAML 条目到末尾
```

## 五、规则配置详解

### 5.1 三层配置

```
~/.bingtangcode/permissions.yaml        ← 用户全局默认（所有项目生效）
<project>/.bingtangcode/permissions.yaml ← 项目共享（可提交 Git）
<project>/.bingtangcode/permissions.local.yaml ← 本地覆盖（不提交 Git）
```

优先级：local > project > user（高优先者覆盖低优先者）

### 5.2 YAML 格式

```yaml
# 可选，启动时的默认模式（不写默认 DEFAULT）
defaultMode: default   # default | acceptEdits | plan | bypassPermissions

rules:
  # Bash 工具：匹配 command 参数
  - tool: Bash
    pattern: "git *"          # glob: git 开头的所有命令
    action: allow

  - tool: Bash
    pattern: "git push --force *"  # deny 优先于 allow
    action: deny

  # 文件工具：匹配 file_path / directory 参数（自动转相对路径）
  - tool: Read
    pattern: ".env"           # 精确: 禁止读 .env
    action: deny

  - tool: Write
    pattern: "*.md"           # glob: 禁止写所有 .md 文件
    action: deny

  - tool: Edit
    pattern: "src/**/*.java"  # glob: 允许编辑 src 下所有 Java 文件
    action: allow

  - tool: Glob
    pattern: "*.java"         # 精确匹配 directory 参数
    action: allow
```

### 5.3 工具名与主参数对应

| 内部名 | 友好名 | 匹配的主参数 |
|--------|--------|-------------|
| `read_file` | Read | file_path |
| `write_file` | Write | file_path |
| `edit_file` | Edit | file_path |
| `execute_command` | Bash | command |
| `find_files` | Glob | directory |
| `search_content` | Grep | directory |

## 六、模式切换

### 6.1 四种切换方式

| 方式 | 触发 | 效果 |
|------|------|------|
| 启动配置 | `permissions.yaml` 中写 `defaultMode: plan` | 启动即生效 |
| `/plan` | 输入 `/plan` | 直接切换到 PLAN |
| `/do` | 输入 `/do` | 切回 DEFAULT |
| `/mode` | 输入 `/mode` | 循环：DEFAULT → ACCEPT_EDITS → PLAN → BYPASS → DEFAULT |
| Shift+Tab | 按 Shift+Tab | 同 `/mode` 循环（终端支持时） |

### 6.2 循环顺序

```
DEFAULT → ACCEPT_EDITS → PLAN → BYPASS_PERMISSIONS → DEFAULT → ...
```

### 6.3 切换后的影响

```
SessionManager.cycleMode()
  ├─ agentLoop.setMode(newMode)      → selectTools() 过滤逻辑更新
  │     └─ PLAN: 只暴露 isReadOnly() == true 的工具
  │     └─ 其他: 暴露全部工具
  ├─ permissionGate.setMode(newMode) → getDefaultAction() 逻辑更新
  └─ System.out.println("已切换到 Plan Mode")
```

### 6.4 Plan 模式的双层设计：硬约束 + 软引导

Plan 模式和其他三个模式有一个根本区别——它在**两个不同层面**同时限制模型行为：

```
Layer A: 工具曝光层（AgentLoop.selectTools）
  └─ API 请求的 tools[] 只放 read_file、find_files、search_content（3 个只读）
  └─ write_file、edit_file、execute_command 根本不出现在请求体中
  └─ 物理上不可绕过——模型想调也调不了，API 不认

Layer B: 系统提醒层（SystemReminderManager）
  └─ 第 1 轮注入完整提醒：Plan模式：仅可用只读工具，专注于分析问题...
  └─ 每 3 轮重复完整提醒
  └─ 其余轮次注入精简版：Plan模式：仅可用只读工具
```

**为什么 Plan 需要两层，其他三个模式不需要：**

| 模式 | 工具曝光 | 系统提醒 | 原因 |
|------|---------|---------|------|
| Default | 全部 6 个 | 无 | 模型看到完整工具集，知道自己的全部能力 |
| AcceptEdits | 全部 6 个 | 无 | 同上 |
| Plan | **仅 3 个只读** | **有（双层注入）** | 模型看到的工具列表是残缺的，不加提醒它会困惑——为什么我只有 3 个工具？是不是系统坏了？ |
| Bypass | 全部 6 个 | 无 | 同上 |

**两层缺一不可：**

| | 只有工具过滤（无提醒） | 只有提醒（无过滤） |
|---|---|---|
| 模型视角 | tools[] 只有 3 个 | tools[] 有 6 个 |
| 模型行为 | 不知道原因，可能认为系统故障，反复重试或编造工具名 | 长对话中可能忽略提醒，直接调 write_file |
| 后果 | 浪费 token，用户困惑 | Plan 模式形同虚设 |

工具过滤是**物理锁**——硬约束，不可绕过。系统提醒是**说明书**——软引导，告诉模型锁在哪、为什么锁、怎么配合。Claude Code 的 Plan 模式用的就是同一套双层设计。

## 七、权限拒绝后的行为

### 7.1 结构化错误格式

```
权限拒绝: Bash(git push --force origin main) 被 匹配黑名单: git\s+push\s+--force\s+.*main 拦截
  匹配规则: blacklist
```

格式：`权限拒绝: <FriendlyName>(<mainParam>) 被 <reason> 拦截\n  匹配规则: <matchedRule>`

### 7.2 拒绝不终止 Agent Loop

```
executeOne(tc)
  └─ permissionGate.check(tc) → DENY
       └─ 构造: ToolResult(tc.id(), "权限拒绝: Bash(git push...) 被 ... 拦截\n  匹配规则: blacklist", isError=true)
       └─ 不调 toolExecutor.execute()  ← 工具根本不执行
       └─ 正常写入 history

AgentLoop 下一轮:
  └─ 模型看到 error ToolResult（含拒因和建议）
  └─ 自主调整策略（换个命令、换个文件、请求用户改规则）
  └─ 继续循环，不触发 CANCELLED
```

**关键点**：
- `hasUnknown` 不会被设为 true（这是权限拒绝，不是未知工具）
- `unknownToolStreak` 不会增加
- 连续多次被拒不会触发 `UNKNOWN_TOOL_LOOP` 终止

## 八、权限检查在哪个线程上

```
Provider IO 线程 (doRound → executeToolBatch):
  ├─ ...
  ├─ executeOne(tc) → permissionGate.check(tc)
  │    ├─ Layer 1~4: 瞬间完成（内存操作）
  │    └─ Layer 5: 阻塞等待用户按键
  │          └─ PermissionPrompt.ask()
  │               └─ while(true) { terminalIO.readKey() }
  │                    阻塞在此直到用户按键 Enter/1/2/3/y/n/d
  ├─ 权限通过 → toolExecutor.execute(tool)
  └─ 权限拒绝 → 构造 error ToolResult

AgentLoop 线程 (session 主线程):
  agentLoop.run() → doRound() → latch.await() 阻塞
  人在回路等待期间它只是等着
```

**注意**：人在回路期间，`terminalIO.readKey()` 会阻塞 Provider IO 线程。这意味着人在回路等待用户按键期间不能发新的 HTTP 请求。但这是合理的——同一个 tool call 还没完成，不可能发下一轮请求。

## 九、与旧系统的对比

| | 旧系统 | 新系统 |
|---|--------|--------|
| 路径检查 | 每个工具自己写 resolve+startsWith | 统一 PathSandbox + 符号链接解析 |
| Bash 确认 | 10 秒超时 y/N | 无限等待箭头选择 |
| 规则配置 | 无 | 三层 YAML |
| 信任沉淀 | 每次都要确认 | ALLOW_FOREVER 写入持久化 |
| 安全等级 | 无分级 | 4 档 PermissionMode |
| 拒绝处理 | 工具返回 error | 结构化错误 + Agent 继续 |
| Plan Mode | AgentLoop 枚举 | 合并到 PermissionMode |
| 模式切换 | 仅 /plan /do | + Shift+Tab + /mode + 配置 |
